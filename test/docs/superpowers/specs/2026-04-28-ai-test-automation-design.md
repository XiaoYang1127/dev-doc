## 一、定位与目标

### 1.1 核心定位

构建**AI 驱动的分层测试自动生成与运行体系**，实现：

- 测试用例三层生成（PRD → 接口 → 代码）
- 接口测试覆盖 10-15% 问题
- 单元测试覆盖 70% 问题
- 全流程自动化运行与报告

### 1.2 目标拆解

| 目标               | 说明                               | 成功指标                           |
| ------------------ | ---------------------------------- | ---------------------------------- |
| **用例自动生成**   | PRD/接口/代码三层来源，AI 辅助生成 | 核心模块生成率 ≥ 80%               |
| **接口测试自动化** | 基于契约自动生成接口测试脚本       | 接口覆盖 100%，问题发现率 15%      |
| **单测自动化**     | 代码提交自动识别并生成单元测试     | 核心逻辑覆盖 ≥ 90%，问题发现率 70% |
| **报告自动化**     | 测试运行、覆盖率、报告全自动化     | 报告生成时间 < 2 分钟              |

### 1.3 解决的问题

- 测试用例准备不充分（遗漏分支、数据不足）
- 测试编写耗时，难以跟上代码生成速度
- 测试与代码不同步（改了代码忘改测试）
- 测试运行分散，难以统一查看结果

### 2.2 模块边界与依赖

```
调用方向：上游 → 下游

PRD/接口/源代码（输入层）
  │
  ▼
AI Generation Engine（AI 生成引擎）
  ├─> PRD 解析器 ──> 生成验收用例
  ├─> 接口解析器 ──> 生成接口测试
  ├─> 代码分析器 ──> 生成单元测试
  └─> 质量评估器 ──> 质量门禁
                           │
                           ▼
              Test Execution Engine（测试执行引擎）
                ├─> 接口测试执行
                └─> 单测执行
                           │
                           ▼
              报告与通知层（输出层）
```

| 模块             | 职责                            | 不管什么       | 输入         | 输出               |
| ---------------- | ------------------------------- | -------------- | ------------ | ------------------ |
| **PRD 解析器**   | 解析 PRD 提取业务规则和边界条件 | 具体测试实现   | PRD 文档     | 验收测试用例 (BDD) |
| **接口解析器**   | 解析 OpenAPI/YAPI 提取契约      | 业务逻辑验证   | OpenAPI 文档 | 接口测试场景       |
| **代码分析器**   | AST 分析识别分支和依赖          | 测试数据准备   | 源代码       | 单元测试代码       |
| **质量评估器**   | 检查覆盖率、可运行性、重复      | 测试业务正确性 | 生成的测试   | 评估报告           |
| **测试执行引擎** | 编译、运行、收集结果            | 测试生成       | 测试代码     | 执行报告           |
| **报告层**       | 聚合结果、生成可视化报告        | 测试执行       | 原始结果     | HTML/JSON 报告     |

## 三、边界职责

### 3.1 AI Generation Engine

```
AI 生成引擎
├── 职责：
│   ├── 解析 PRD/接口/代码三种输入
│   ├── 生成对应类型的测试用例
│   ├── 评估生成质量（覆盖、可运行）
│   └── 与执行引擎交互获取反馈
├── 不管：
│   ├── 具体的测试执行
│   ├── 测试环境的部署
│   └── 报告的展示渲染
├── 被谁调用：
│   ├── CI/CD 流水线
│   ├── IDE 插件
│   └── 测试中台
└── 调用谁：
    ├── LLM API（生成用例）
    ├── 代码仓库（读取源码）
    └── 测试执行引擎（验证可运行性）
```

### 3.2 Test Execution Engine

```
测试执行引擎
├── 职责：
│   ├── 编译测试代码
│   ├── 运行测试套件
│   ├── 收集覆盖率数据
│   └── 返回执行结果
├── 不管：
│   ├── 测试用例的生成
│   ├── 质量门禁的判断
│   └── 通知的发送
├── 被谁调用：
│   ├── AI Generation Engine
│   ├── CI/CD 流水线
│   └── 测试中台
└── 调用谁：
    ├── Maven/Gradle（编译运行）
    ├── TestContainers（容器环境）
    └── JaCoCo（覆盖率）
```

### 3.3 报告与通知层

```
报告与通知层
├── 职责：
│   ├── 聚合多维度测试结果
│   ├── 生成可视化报告
│   ├── 质量门禁判定
│   └── 多渠道通知推送
├── 不管：
│   ├── 测试的生成
│   ├── 测试的执行
│   └── 业务逻辑验证
├── 被谁调用：
│   └── 测试执行引擎
└── 调用谁：
    ├── Allure（报告渲染）
    ├── GitLab/GitHub API（状态回写）
    └── 钉钉/企微 API（通知）
```

### 4.2 详细流程：PRD 验收用例生成

```
触发条件：PRD 评审完成

阶段 1：PRD 解析
├── 输入：PRD 文档（Markdown/Word）
├── AI 任务：
│   ├── 识别功能模块
│   ├── 提取业务规则（Given-When-Then）
│   ├── 提取边界条件
│   └── 提取异常场景
└── 输出：结构化需求列表

阶段 2：用例生成
├── 输入：结构化需求
├── AI 任务：
│   ├── 生成 BDD 特征文件
│   ├── 生成测试数据
│   └── 生成步骤定义模板
└── 输出：.feature 文件 + 步骤定义骨架

阶段 3：人工审核
├── 责任人：QA/产品
├── 任务：
│   ├── 审核用例完整性
│   ├── 补充特殊场景
│   └── 确认验收标准
└── 输出：最终验收用例

存储位置：tests/acceptance/*.feature
```

### 4.3 详细流程：接口测试生成与运行

```
触发条件：接口设计完成（OpenAPI/YAPI）

阶段 1：接口解析
├── 输入：OpenAPI 3.0 文档 / YAPI 导出
├── AI 任务：
│   ├── 解析 Paths 和 Methods
│   ├── 提取参数约束（必填、类型、范围）
│   ├── 提取响应结构
│   └── 识别状态码含义
└── 输出：接口契约模型

阶段 2：场景生成
├── 输入：接口契约模型
├── AI 任务：
│   ├── 生成正常场景（200 OK）
│   ├── 生成参数校验场景（400 Bad Request）
│   ├── 生成鉴权场景（401/403）
│   ├── 生成边界值场景
│   └── 生成业务规则场景
└── 输出：测试场景 YAML

阶段 3：脚本生成
├── 输入：测试场景
├── AI 任务：
│   ├── 生成 REST Assured 测试代码
│   ├── 生成测试数据（Faker）
│   ├── 生成 Mock 配置（WireMock）
│   └── 生成断言逻辑
└── 输出：Java 测试类

阶段 4：自动运行
├── 触发：MR 创建 / 定时任务
├── 执行：
│   ├── 启动测试环境（TestContainers）
│   ├── 运行接口测试集
│   ├── 收集覆盖率
│   └── 生成 Allure 报告
└── 结果：通过/失败 + 报告链接

存储位置：tests/api/*.java
```

```
│   ├── 生成 Mock 配置（Mockito）
│   ├── 生成测试方法（正例/反例/边界）
│   ├── 生成参数化测试（@CsvSource）
│   └── 生成断言（AssertJ）
└── 输出：Java 测试代码

阶段 3：质量评估
├── 自动检查：
│   ├── 编译检查
│   ├── 运行检查
│   ├── 覆盖率检查（增量）
│   └── 重复检测
├── 结果判定：
│   ├── 全部通过 → 自动提交 Review
│   └── 部分失败 → 人工审核

阶段 4：CI 运行
├── 触发：代码 Push
├── 执行：
│   ├── mvn test
│   ├── JaCoCo 覆盖率
│   └── 增量覆盖报告
└── 门禁：
    ├── 核心模块 ≥ 90% 行覆盖
    ├── 整体 ≥ 70% 行覆盖
    └── 测试通过率 ≥ 95%

存储位置：src/test/java/**/*Test.java
```

### 4.5 失败处理策略

| 场景              | 处理策略                                        | 责任人    |
| ----------------- | ----------------------------------------------- | --------- |
| AI 生成用例质量低 | 人工审核 → 反馈优化 Prompt → 重新生成           | QA + 开发 |
| 测试编译失败      | AI 自动修复（3 次）→ 人工介入                   | AI/开发   |
| 测试运行失败      | 分析失败类型 → 代码 Bug 则修复 → 测试问题则更新 | 开发      |
| 覆盖率不达标      | 识别未覆盖代码 → AI 补充测试 → 人工审核         | AI/开发   |
| 接口契约变更      | 重新生成接口测试 → 对比差异 → 人工确认          | AI/QA     |
