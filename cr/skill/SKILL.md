# 代码审核助手

## 描述

针对 Java 项目的代码变更进行分层审核，专注于**架构合规性**和**明显缺陷**的检测。

- **当前阶段**：宽松建议型，只标记高价值问题
- **完整规则**：见 [design/rules/languages/java.md](../design/rules/languages/java.md)
- **审核规范**：见 [java-review.md](./java-review.md)

## 触发条件

用户输入：
- `/cr` - 审核当前 git 变更
- `/cr <文件路径>` - 审核指定文件
- `/cr --strict` - 启用严格模式（未来支持）

## 执行流程

### Step 1: 获取变更

```bash
git diff --cached --name-only          # 暂存区变更
git diff HEAD --name-only             # 工作区变更
```

过滤出 `.java` 文件，其他语言提示暂未支持。

### Step 2: 加载上下文

对于每个变更文件：
1. 读取完整文件内容
2. 识别所在模块（根据 package 路径）
3. 识别变更类型（新增/修改/删除）

### Step 3: 分层审核

#### Layer 1: 分层架构检查（最高优先级）

检查项：
- [ ] Controller 是否直接调用 Repository/Mapper
- [ ] Service 是否直接操作其他模块的 Repository
- [ ] PO 是否直接返回给前端（未通过 DTO 转换）

#### Layer 2: 事务安全（第二优先级）

检查项：
- [ ] @Transactional 方法内是否有 HTTP/RPC 调用
- [ ] @Transactional 方法内是否有异步操作（@Async/MQ 发送）
- [ ] 事务传播行为是否合理（默认 REQUIRED 是否足够）

#### Layer 3: NPE 风险（第三优先级）

检查项：
- [ ] 方法参数未校验直接使用（如 dto.getName() 可能 NPE）
- [ ] 集合操作前未判空（list.stream() 可能 NPE）
- [ ] Optional.get() 未先判空

#### Layer 4: 明显逻辑缺陷（第四优先级）

检查项：
- [ ] 死代码（unreachable code）
- [ ] 资源未关闭（Stream/Connection 等）
- [ ] 错误的比较方式（== 比较对象、浮点数直接比较）

### Step 4: 输出报告

格式要求：

```markdown
## 代码审核报告

### 概览
- 审核文件：{n} 个
- 发现问题：严重 {x} | 警告 {y} | 建议 {z}

---

### {文件名}:{行号}

**等级**: 🔴 严重 / 🟡 警告 / 💡 建议
**类型**: 分层越界 / 事务安全 / NPE风险 / 逻辑缺陷
**问题**: 一句话描述
**代码**:
```java
[变更代码片段]
```
**建议**:
```java
[修复建议]
```
**理由**: 为什么这是个问题

---

[下一个问题...]
```

## 审核原则

1. **宁可漏报，不可误报** - 宽松阶段，确保每条问题都是真问题
2. **给出修复代码** - 不只说"错了"，要说"怎么改"
3. **解释为什么** - 帮助开发者理解问题本质
4. **不纠结风格** - 暂时不管命名、格式、注释等风格问题
