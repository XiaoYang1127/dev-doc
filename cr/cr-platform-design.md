# 代码审核中台架构设计

**日期**：2026-03-29
**状态**：草稿

---

## 一、定位与目标

### 1.1 核心定位

代码审核中台是**智能代码审核的平台化基础设施**，基于 10 层审核架构，为各业务团队提供统一的代码审核能力，支持从个人本地审核到团队 CI 集成的完整场景。

### 1.2 目标

- 统一审核入口，标准化审核维度
- AI 辅助审核，降低人工审核成本
- 可配置审核策略，支持不同场景
- 持续学习优化，提升审核准确率
- 与代码仓库（GitLab）深度集成

### 1.3 设计原则

- **平台侧**：负责审核引擎、规则管理、结果处理、学习闭环
- **业务侧**：可自定义审核规则，平台提供规则框架
- **审核边界**：AI 审核模式化问题，人工审核复杂业务逻辑

### 2.2 模块边界与依赖

```
调用方向：上游 → 下游

Gateway（接入层）
  └─> Core Engine（核心调度层）
        ├─> 变更审核引擎（1-8 层）
        ├─> 全量巡检引擎（L9）
        └─> 反馈学习引擎（L10）
              │
              └─> Rule System（规则系统）
```

| 模块         | 职责                         | 不管什么           |
| ------------ | ---------------------------- | ------------------ |
| Gateway      | 事件接入、协议适配           | 审核逻辑、规则执行 |
| Core Engine  | 触发调度、策略路由、结果聚合 | 具体审核实现       |
| 变更审核引擎 | L1-L8 审核执行               | 全量巡检、规则管理 |
| 全量巡检引擎 | L9 代码库健康度检查          | 增量变更审核       |
| 反馈学习引擎 | L10 反馈闭环、规则优化       | 审核执行           |
| Rule System  | 规则解析、执行、版本管理     | 具体业务审核       |

### 2.3 设计原则

| 原则                 | 说明                               |
| -------------------- | ---------------------------------- |
| **单向依赖**         | 模块只能调用下游模块，不能反向调用 |
| **Core Engine 自主** | 不被具体审核实现依赖，可独立演进   |
| **Rule System 底层** | 作为基础层，被各审核引擎共用       |
| **Gateway 隔离**     | 不感知内部模块，只负责事件接入     |

## 三、核心模块详解

### 3.1 Gateway（平台接入层）

| 职责     | 说明                                               |
| -------- | -------------------------------------------------- |
| 事件接收 | 接收上游 Webhook 事件（MR 创建、更新、定时触发等） |
| 协议适配 | 将上游事件转换为平台内部事件格式                   |
| 认证鉴权 | 验证请求合法性                                     |

**边界要点**：平台不感知上游是 GitLab 还是 GitHub，适配层负责协议转换。

### 3.2 Core Engine（核心引擎）

| 职责     | 说明                           |
| -------- | ------------------------------ |
| 触发调度 | 根据触发条件分发审核任务       |
| 策略路由 | 根据配置选择审核维度组合       |
| 结果聚合 | 收集各审核层结果，生成统一报告 |

**边界要点**：核心引擎**不**执行具体审核逻辑，只负责任务调度和结果聚合。

### 3.3 变更审核引擎

基于代码变更的实时审核，是最核心的审核场景。

| 层级 | 维度       | 说明                  | 复杂度 |
| ---- | ---------- | --------------------- | ------ |
| L1   | 变更感知   | git diff 获取变更信息 | 低     |
| L2   | 片段审核   | 只看变更代码本身      | 低     |
| L3   | 上下文审核 | 变更所在函数完整分析  | 中     |
| L4   | 架构审核   | 分层合规性检查        | 中     |
| L5   | 调用链审核 | 上下游调用关系分析    | 高     |
| L6   | 数据流审核 | 事务边界、并发安全    | 高     |
| L7   | 安全审核   | 注入攻击、越权访问    | 中     |
| L8   | 性能审核   | N+1 查询、性能隐患    | 中     |

### 3.4 全量巡检引擎（L9）

| 职责         | 说明                         |
| ------------ | ---------------------------- |
| 定时扫描     | 每日/每周全量代码健康度检查  |
| 架构腐化检测 | 分层违规趋势、依赖方向       |
| 技术债务追踪 | 圈复杂度、重复代码、废弃 API |

### 3.5 反馈学习引擎（L10）

| 职责     | 说明                           |
| -------- | ------------------------------ |
| 反馈收集 | 收集开发者对审核意见的处理结果 |
| 规则优化 | 基于反馈调整规则阈值、更新规则 |
| A/B 测试 | 新规则小流量验证               |

### 3.6 Rule System（规则系统）

#### 3.6.1 规则类型

| 类型     | 说明           | 示例                   |
| -------- | -------------- | ---------------------- |
| DSL 规则 | 声明式规则配置 | 覆盖率阈值、复杂度限制 |
| AST 规则 | 语法树模式匹配 | 分层越界、事务边界     |
| AI 规则  | AI 辅助分析    | 代码逻辑缺陷、业务问题 |

#### 3.6.2 规则结构

```java
Rule {
    String ruleId;           // 规则唯一标识
    String name;             // 规则名称
    RuleType type;           // DSL / AST / AI
    Severity severity;        // 严重 / 警告 / 建议
    List<Layer> appliesTo;   // 适用的审核层级
    String expression;        // 规则表达式
    String suggestion;       // 修复建议
}
```

## 四、工作流程

### 4.1 MR 审核主流程

```
触发阶段
│
├─ 1. MR 创建 / 更新 / 代码提交
├─ 2. Gateway 接收 Webhook 事件
├─ 3. 协议转换 → 内部 ReviewEvent
└─ 4. Core Engine 创建 ReviewPipeline
        │
        ▼
变更审核阶段
│
├─ 5. 变更审核引擎执行 L1-L8
│   ├─ L1: 变更感知（git diff）
│   ├─ L2: 片段审核（变更本身）
│   ├─ L3: 上下文审核（完整函数）
│   ├─ L4: 架构审核（分层合规）
│   ├─ L5: 调用链审核（影响分析）
│   ├─ L6: 数据流审核（事务/并发）
│   ├─ L7: 安全审核（注入/越权）
│   └─ L8: 性能审核（N+1/资源）
│
        ▼
结果处理阶段
│
├─ 6. 结果聚合
│   ├─ 去重（基于 location + rule_id）
│   ├─ 分级（严重 / 警告 / 建议）
│   └─ 排序（严重度 > 置信度）
│
├─ 7. 存储审核结果
└─ 8. 生成审核报告
        │
        ▼
回执阶段
│
├─ 9. 回执给上游
│   ├─ GitLab MR 评论
│   └─ GitLab Status Check
└─ 10. 通知（钉钉/企微/邮件）
```

#### 4.1.1 触发策略

| 场景     | 触发时机   | 执行层级      | 说明       |
| -------- | ---------- | ------------- | ---------- |
| 本地开发 | 手动 `/cr` | L1-L4         | 快速反馈   |
| MR 创建  | Webhook    | L1-L8         | 完整审核   |
| MR 更新  | 新 commit  | L1-L8（增量） | 只审变更   |
| 定时任务 | 每晚       | L9 全量       | 健康度检查 |

### 4.2 全量巡检流程

```
定时触发 / 手动触发
        │
        ▼
全量巡检引擎执行 L9
        │
        ▼
代码健康度检查
├─ 圈复杂度
├─ 重复代码
├─ 架构违规趋势
└─ 测试覆盖率
        │
        ▼
生成健康度报告
        │
        ▼
通知（周报 / 仪表盘）
```

### 4.3 反馈学习流程

```
审核意见发出
        │
        ▼
开发者反馈
├─ 接受（已修复）
├─ 拒绝（误报）
└─ 忽略
        │
        ▼
反馈学习引擎收集
        │
        ▼
定期分析
├─ 误报率 > 30% → 规则下线/调优
├─ 漏报案例 → 补充规则
└─ 高频问题 → 优化提示词
        │
        ▼
规则更新
        │
        ▼
A/B 测试验证
        │
        ▼
全量发布
```

## 五、边界定义

### 5.1 AI 审核 vs 人工审核

| 维度         | AI 审核                          | 人工审核               |
| ------------ | -------------------------------- | ---------------------- |
| **发现缺陷** | 模式化问题（分层越界、事务风险） | 复杂业务逻辑、设计决策 |
| **效率**     | 秒级响应                         | 耗时且容易疲劳         |
| **一致性**   | 标准化，不因人而异               | 因经验/状态而异        |
| **业务理解** | ❌                               | ✅                     |
| **架构权衡** | 部分 ✅                          | ✅                     |

### 5.2 审核边界

| AI 审核做            | AI 审核不做      |
| -------------------- | ---------------- |
| 分层合规性检查       | 代码风格审美判断 |
| 事务边界检查         | 注释质量评估     |
| NPE 风险识别         | 方法长度硬限制   |
| 安全漏洞扫描（基础） | 架构设计决策     |
| 明显逻辑缺陷         | 业务逻辑正确性   |
| N+1 查询检测         | 业务性能基准     |

### 5.3 平台与业务边界

| 平台做              | 业务方可扩展   |
| ------------------- | -------------- |
| 10 层审核架构       | 自定义审核规则 |
| 通用规则库（46 条） | 业务特定规则   |
| 触发调度引擎        | 审核策略配置   |
| 结果聚合与通知      | 通知渠道定制   |
| 反馈学习框架        | 规则阈值调整   |

## 六、审核结果管理

### 6.1 结果数据结构

```java
ReviewIssue {
    String issueId;              // 唯一标识
    String source;               // 来源：L2/L3/L4/L5/L6/L7/L8
    Severity severity;            // 严重 / 警告 / 建议
    Category category;            // 架构 / 安全 / 性能 / 逻辑
    Location location;            // file, line, method, class
    String description;            // 问题描述
    String suggestion;            // 修复建议
    String codeSnippet;            // 当前代码
    String fixedCode;              // 建议修改后的代码
    String ruleId;                 // 关联规则 ID
    float confidence;             // AI 置信度
    IssueStatus status;            // 待处理 / 已确认 / 已修复 / 误报
    Long createdAt;
}

enum Severity {
    CRITICAL,   // 必须修复
    WARNING,    // 建议修复
    SUGGESTION  // 可选优化
}

enum Category {
    ARCHITECTURE,  // 分层越界
    TRANSACTION,   // 事务安全
    NPE,           // 空指针风险
    SECURITY,       // 安全漏洞
    PERFORMANCE,    // 性能隐患
    LOGIC           // 逻辑缺陷
}
```

### 6.2 状态机

```
待处理(PENDING)
    │
    ├─ 开发者确认 → 已确认(CONFIRMED)
    │       │
    │       └─ 修复完成 → 已修复(FIXED)
    │
    ├─ 标记误报 → 误报(FALSE_POSITIVE)
    │
    └─ 忽略 → 忽略(IGNORED)
```

## 七、演进路线

```
Phase 1: 个人本地（当前）
   └── Java 宽松审核（L1-L4 核心规则）

Phase 2: 个人本地增强
   ├── Java 严格审核（开启 L7-L8）
   ├── Vue/Flutter 审核
   └── 本地 CLI 工具

Phase 3: 团队 CI 集成
   └── GitLab MR 自动审核 + 评论通知

Phase 4: 持续学习
   └── 反馈闭环 + 规则自动优化（L10）
```
