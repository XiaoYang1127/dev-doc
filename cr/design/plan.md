## 1. 系统架构概览

### 1.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        代码审核系统架构                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────────┐    │
│   │   触发引擎   │───▶│   调度引擎   │───▶│   执行引擎       │    │
│   │  (Trigger)  │    │ (Scheduler) │    │   (Executor)    │    │
│   └─────────────┘    └─────────────┘    └─────────────────┘    │
│                                                │                │
│                    ┌───────────────────────────┼───────────┐    │
│                    ▼                           ▼           ▼    │
│              ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌────────┐ │
│              │片段审核 │  │上下文审核│  │架构审核 │  │安全审核│ │
│              │(轻量)   │  │(中等)   │  │(中等)   │  │(轻量)  │ │
│              └─────────┘  └─────────┘  └─────────┘  └────────┘ │
│              ┌─────────┐  ┌─────────┐  ┌─────────┐            │
│              │调用链审核│  │数据流审核│  │性能审核 │            │
│              │(较重)   │  │(较重)   │  │(中等)   │            │
│              └─────────┘  └─────────┘  └─────────┘            │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                      结果管理层                          │  │
│   │  去重 → 分级 → 存储 → 通知 → 追踪 → 统计                 │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                      反馈学习层                          │  │
│   │  反馈收集 → 规则评估 → 阈值调优 → 规则更新 → A/B测试      │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 核心设计原则

| 原则         | 说明                               |
| ------------ | ---------------------------------- |
| **分层解耦** | 每个审核维度独立，可单独启用/禁用  |
| **渐进增强** | 从简单规则开始，逐步叠加复杂分析   |
| **可配置**   | 不同场景选择不同审核维度组合       |
| **可度量**   | 每条审核结果可追踪、可反馈、可优化 |

## 2. 审核维度设计

### 维度1：变更感知层

| 项目           | 内容                                                                         |
| -------------- | ---------------------------------------------------------------------------- |
| **话题**       | 变更感知与获取                                                               |
| **变更**       | 从Git获取代码变更的基础信息                                                  |
| **审核侧重点** | 识别变更范围、变更类型、影响文件                                             |
| **实现做法**   | ① `git diff` 获取行级差异 ② `git status` 获取文件状态 ③ 解析diff提取变更行号 |
| **审核要求**   | ① 识别新增/修改/删除 ② 标记变更边界 ③ 过滤二进制/非代码文件                  |

### 维度2：片段审核层

| 项目           | 内容                                                                                                 |
| -------------- | ---------------------------------------------------------------------------------------------------- |
| **话题**       | 变更片段独立审核                                                                                     |
| **变更**       | 仅看变更代码本身，不看上下文                                                                         |
| **审核侧重点** | 语法错误、明显缺陷、安全漏洞                                                                         |
| **实现做法**   | ① 提取diff中的 `+` 代码块 ② Claude直接分析 ③ 正则规则快速筛选                                        |
| **审核要求**   | ① 语法合法性 ② 常见反模式（`==` 比较对象、空指针链式调用）③ 敏感信息泄露（密钥、密码）④ 硬编码字符串 |

### 维度3：上下文审核层

| 项目           | 内容                                                                                   |
| -------------- | -------------------------------------------------------------------------------------- |
| **话题**       | 变更上下文关联审核                                                                     |
| **变更**       | 获取变更所在函数的完整代码                                                             |
| **审核侧重点** | 变更与上下文的逻辑一致性                                                               |
| **实现做法**   | ① 解析diff行号 ② 读取完整函数/方法 ③ 分析控制流和数据流                                |
| **审核要求**   | ① 变更是否破坏原有逻辑 ② 边界条件处理 ③ 异常处理完整性 ④ 变量作用域冲突 ⑤ 返回值一致性 |

### 维度4：架构审核层

| 项目           | 内容                                                                                                                |
| -------------- | ------------------------------------------------------------------------------------------------------------------- |
| **话题**       | 分层架构合规性审核                                                                                                  |
| **变更**       | 检查代码是否符合项目架构规范                                                                                        |
| **审核侧重点** | 职责边界、依赖方向、数据流转                                                                                        |
| **实现做法**   | ① 正则匹配类名识别层级 ② 检查跨层调用 ③ 验证DTO/PO转换                                                              |
| **审核要求**   | ① Controller是否直接调Repository ② 是否跨模块操作数据 ③ PO是否直接返回前端 ④ Service是否过于臃肿 ⑤ 是否违反依赖方向 |

### 维度5：调用链审核层

| 项目           | 内容                                                                                                                                  |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **话题**       | 全局影响分析                                                                                                                          |
| **变更**       | 分析变更方法的上下游调用关系                                                                                                          |
| **审核侧重点** | 变更的级联影响、兼容性                                                                                                                |
| **实现做法**   | ① AST解析获取方法签名 ② 全局搜索调用点 ③ 构建调用链图                                                                                 |
| **审核要求**   | ① 上游调用者是否受影响 ② 下游被调者行为是否变化 ③ 接口兼容性（参数变化、返回值变化）④ 是否需要同步修改其他模块 ⑤ 废弃方法是否仍有调用 |

### 维度6：数据流审核层

| 项目           | 内容                                                                                               |
| -------------- | -------------------------------------------------------------------------------------------------- |
| **话题**       | 数据流转与状态变更审核                                                                             |
| **变更**       | 追踪数据从输入到输出的完整路径                                                                     |
| **审核侧重点** | 数据一致性、事务边界、并发安全                                                                     |
| **实现做法**   | ① 识别 `@Transactional` 边界 ② 分析数据库操作顺序 ③ 检查异步操作                                   |
| **审核要求**   | ① 事务内是否有远程调用 ② 事务内是否有MQ发送 ③ 竞态条件风险 ④ 数据一致性保障 ⑤ 死锁风险（多锁顺序） |

### 维度7：安全审核层

| 项目           | 内容                                                                                                                    |
| -------------- | ----------------------------------------------------------------------------------------------------------------------- |
| **话题**       | 安全漏洞检测                                                                                                            |
| **变更**       | 识别常见的安全缺陷                                                                                                      |
| **审核侧重点** | 注入攻击、越权访问、敏感信息                                                                                            |
| **实现做法**   | ① 正则+AI混合检测 ② 敏感信息模式匹配 ③ 调用链分析识别危险函数                                                           |
| **审核要求**   | ① SQL注入风险（字符串拼接SQL）② XSS风险（未转义输出）③ 硬编码密钥/密码 ④ 不安全的反序列化 ⑤ 越权检查缺失 ⑥ 敏感日志输出 |

### 维度8：性能审核层

| 项目           | 内容                                                                                                                 |
| -------------- | -------------------------------------------------------------------------------------------------------------------- |
| **话题**       | 性能隐患检测                                                                                                         |
| **变更**       | 识别可能导致性能问题的代码                                                                                           |
| **审核侧重点** | 算法复杂度、资源消耗、高频操作                                                                                       |
| **实现做法**   | ① 代码模式识别 ② 嵌套循环检测 ③ 数据库操作分析                                                                       |
| **审核要求**   | ① N+1查询问题 ② 大对象创建（大List、大JSON）③ 循环内数据库操作 ④ 不当的同步锁 ⑤ 内存泄漏风险（未释放资源）⑥ 低效算法 |

### 维度9：全量扫描层

| 项目           | 内容                                                                                         |
| -------------- | -------------------------------------------------------------------------------------------- |
| **话题**       | 代码库整体健康度检查                                                                         |
| **变更**       | 不针对变更，针对整个代码库                                                                   |
| **审核侧重点** | 架构腐化、技术债务、规范符合度                                                               |
| **实现做法**   | ① 结合SonarQube等工具 ② 定期全量AST分析 ③ 趋势分析（债务增长）                               |
| **审核要求**   | ① 圈复杂度超标 ② 重复代码块 ③ 废弃API使用 ④ 架构规则违反趋势 ⑤ 测试覆盖率变化 ⑥ 代码异味累积 |

### 维度10：反馈学习层

| 项目           | 内容                                                                                 |
| -------------- | ------------------------------------------------------------------------------------ |
| **话题**       | 审核系统的持续优化                                                                   |
| **变更**       | 基于人工反馈改进审核质量                                                             |
| **审核侧重点** | 减少误报、发现漏报、完善规则                                                         |
| **实现做法**   | ① 收集开发者反馈（标记误报/确认有效）② 定期分析审核数据 ③ 更新规则库                 |
| **审核要求**   | ① 误报率统计 ② 漏报案例分析 ③ 规则有效性评分 ④ 审核阈值动态调整 ⑤ 审核维度优先级优化 |
| **详细实现**   | 见 [implementation.md#5-反馈学习层-l10](./implementation.md#5-反馈学习层-l10)         |

## 3. 触发策略

### 3.1 触发场景矩阵

| 场景     | 触发时机        | 执行维度          | 原因               |
| -------- | --------------- | ----------------- | ------------------ |
| 本地开发 | 手动触发 `/cr`  | 1-4层             | 快速反馈，轻量检查 |
| 提交前   | pre-commit hook | 1-3层             | 拦截明显问题       |
| MR创建   | GitLab Webhook  | 1-6层             | 完整审核，全面把关 |
| MR更新   | 新commit推送    | 1-6层（增量）     | 只审变更部分       |
| 定时任务 | 每晚            | 全量扫描（第9层） | 发现架构腐化       |

### 3.2 动态策略配置

```yaml
# 审核策略配置示例
triggers:
  local:
    enabled_layers: [1, 2, 3, 4]
    max_file_size: 100KB
    skip_patterns: ["*.test.java", "*Test.java"]

  pre_commit:
    enabled_layers: [1, 2, 3]
    fail_on_severity: error
    timeout_seconds: 30

  mr_created:
    enabled_layers: [1, 2, 3, 4, 5, 6]
    include_call_chain: true
    max_depth: 3

  nightly:
    enabled_layers: [9]
    full_ast_analysis: true
    trend_analysis: true
```

## 4. 结果管理

### 4.1 审核结果数据模型

```yaml
ReviewIssue:
  id: UUID # 唯一标识，用于去重和追踪
  source: string # 来源：片段/上下文/架构/调用链/数据流/安全/性能
  severity: enum # 严重/警告/建议
  category: enum # 分类：架构/安全/性能/逻辑/NPE/...
  location:
    file: string
    line_start: int
    line_end: int
    method: string # 所在方法
    class: string # 所在类
  description: string # 问题描述
  suggestion: string # 修复建议
  code_snippet: string # 当前代码
  fixed_code: string # 建议修改后的代码
  rule_id: string # 关联的规则ID，用于统计和调优
  confidence: float # AI置信度，用于过滤
  status: enum # 待处理/已确认/已修复/误报/忽略
  created_at: timestamp
  resolved_at: timestamp

  feedback: # 开发者反馈
    is_valid: boolean
    comment: string
    reviewer: string
    reviewed_at: timestamp
```

### 4.2 结果处理流程

```
审核结果
    │
    ▼
┌─────────────┐
│   去重      │  基于 location + rule_id + code_hash
└─────────────┘
    │
    ▼
┌─────────────┐
│   分级      │  严重(必须修复) / 警告(建议修复) / 建议(可选)
└─────────────┘
    │
    ▼
┌─────────────┐
│   排序      │  严重度 > 置信度 > 文件路径
└─────────────┘
    │
    ▼
┌─────────────┐
│   存储      │  数据库/文件/缓存
└─────────────┘
    │
    ▼
┌─────────────┐
│   通知      │  GitLab评论/邮件/IM消息
└─────────────┘
    │
    ▼
┌─────────────┐
│   追踪      │  状态变更追踪、修复验证
└─────────────┘
```

### 4.3 输出格式规范


## 5. 反馈学习

### 5.1 反馈收集机制

| 反馈类型 | 收集方式                     | 用途             |
| -------- | ---------------------------- | ---------------- |
| 误报标记 | GitLab评论 `/false-positive` | 降低该类问题权重 |
| 确认有效 | 开发者按原建议修复           | 提高该类问题权重 |
| 忽略标记 | GitLab评论 `/ignore`         | 加入白名单       |
| 补充说明 | 评论补充上下文               | 优化审核提示词   |

### 5.2 规则优化流程

```

收集反馈 (每周)
│
▼
统计分析
├─ 误报率 > 30% 的规则 → 调优或下线
├─ 漏报案例 → 补充检测规则
└─ 高频问题 → 生成示例代码
│
▼
规则更新
├─ 调整正则/AST匹配模式
├─ 优化AI提示词
└─ 更新审核阈值
│
▼
A/B测试
├─ 新规则小流量试点
├─ 对比误报率变化
└─ 全量发布或回滚

```

### 5.3 持续学习的数据沉淀

```

审核历史库
├─ 已确认问题 → 正样本
├─ 误报标记 → 负样本
├─ 修复方案 → 知识库
└─ 趋势数据 → 架构健康度报告

```

## 6. 多语言支持

### 6.1 分层抽象架构

```

┌─────────────────────────────────────────┐
│ 审核规则层 (Rules) │ ← 语言无关的业务规则
│ (分层违规、事务安全、NPE风险) │
├─────────────────────────────────────────┤
│ 语义分析层 (Semantic) │ ← 跨语言的通用概念
│ (函数、类、调用关系、控制流、数据流) │
├─────────────────────────────────────────┤
│ AST解析层 (Parser) │ ← 语言特定
│ Java: JavaParser/Spoon │
│ Dart: dart_analyzer │
│ TS/Vue: TypeScript Compiler/Babel │
├─────────────────────────────────────────┤
│ 源码获取层 (Source) │ ← 通用
│ (git diff, file read) │
└─────────────────────────────────────────┘

```

### 6.2 语言支持路线图

| 优先级 | 语言           | 解析工具             | 支持维度     |
| ------ | -------------- | -------------------- | ------------ |
| P0     | Java           | JavaParser / Spoon   | 全部10个维度 |
| P1     | TypeScript/Vue | TS Compiler API      | 1-4, 7-8维度 |
| P1     | Dart(Flutter)  | dart_analyzer        | 1-4, 7-8维度 |
| P2     | Python         | ast / tree-sitter    | 1-4, 7-8维度 |
| P2     | Go             | go/ast / tree-sitter | 1-4, 7-8维度 |

## 7. 实施阶段

### 7.1 阶段划分

| 阶段        | 目标            | 时间 | 交付物             |
| ----------- | --------------- | ---- | ------------------ |
| **Phase 1** | 片段+上下文审核 | 2周  | 本地脚本、MR机器人 |
| **Phase 2** | 架构审核        | 2周  | 分层规则库         |
| **Phase 3** | 调用链+数据流   | 4周  | AST分析引擎        |
| **Phase 4** | 安全+性能       | 3周  | 专项检测规则       |
| **Phase 5** | 全量扫描+学习   | 4周  | 数据平台、反馈闭环 |

### 7.2 成功指标

| 指标         | Phase 1目标  | 最终目标     |
| ------------ | ------------ | ------------ |
| 审核覆盖率   | 变更文件100% | 全量代码100% |
| 误报率       | < 30%        | < 10%        |
| 漏报率       | < 20%        | < 5%         |
| 平均审核时间 | < 60s        | < 30s        |
| 开发者满意度 | > 60%        | > 85%        |

### 7.3 风险与应对

| 风险           | 影响          | 应对策略                                 |
| -------------- | ------------- | ---------------------------------------- |
| AST解析性能差  | 审核时间过长  | ① 增量解析 ② 缓存AST ③ 异步执行          |
| 误报过多       | 开发者不信任  | ① 保守阈值 ② 快速反馈闭环 ③ 白名单机制   |
| 多语言支持复杂 | 开发周期长    | ① 先Java验证 ② 提取通用层 ③ 社区工具复用 |
| 大MR处理困难   | 超时/内存溢出 | ① 分片处理 ② 抽样审核 ③ 仅审关键文件     |

## 附录

### A. 审核维度快速对照表

| 维度       | 输入           | 输出           | 复杂度 | 依赖       |
| ---------- | -------------- | -------------- | ------ | ---------- |
| 变更感知   | git diff       | 变更列表       | 低     | 无         |
