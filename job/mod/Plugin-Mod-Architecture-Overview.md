## 一、核心思路

### 1.1 概念定义

任何插件化系统，架构上都只有**三个实体**和**三个衔接面**：

```
  插件（Plugin）                        ← 实体 2
     │
     ├─ Manifest ── "我是谁、插在哪、什么时候启动"
     │      ▲                                           ▲
     │      │  衔接面 1：Manifest（声明）                 │
     │      │  插件通过声明文件告诉本体自己有什么            │
     │      │                                           │
     │      ▼  衔接面 2：扩展点（Slots）                  │
     │      │  本体定义插槽，插件往里填                     │
     ▼      ▼                                           │
  本体（Core）◄─────────┘                              ← 实体 1
     │
     │  衔接面 3：编程接口（API）
     │  本体暴露方法，插件通过 import/require 调用
     │
     ▼
  Host（运行环境）                       ← 实体 3
     插件在这里被加载、激活、隔离、卸载
```

|                      | 是什么                                                                                                                     | 一句话                       | 以 VSCode 为例                                     |
| -------------------- | -------------------------------------------------------------------------------------------------------------------------- | ---------------------------- | -------------------------------------------------- |
| **本体（Core）**     | 实体。稳定不动的基础框架——定义扩展点、提供编程接口、管理插件生命周期                                                       | "没有插件照样能跑的壳"       | 编辑器窗口、命令面板、菜单栏                       |
| **插件（Plugin）**   | 实体。挂载到本体上的独立功能包——有自己的 Manifest、入口代码、激活条件                                                      | "挂到框架上才有用的能力包"   | Python 支持、GitLens——每个是独立包                 |
| **Host（运行环境）** | 实体。插件代码实际执行的地方——决定隔离强度、故障半径、通信方式                                                             | "插件住的房间，崩了不炸房子" | Extension Host 独立进程                            |
| **Manifest**         | 衔接面 ①。插件→本体的声明通道。插件通过配置文件告诉本体"我是谁、挂哪个插槽、何时激活"，本体不运行插件代码就能读到          | 插件递出的"身份证"           | `package.json` 的 `contributes`+`activationEvents` |
| **扩展点（Slot）**   | 衔接面 ②。本体→插件的接入通道。本体预留固定位置，插件只能往这些位置填，不能随意注入                                        | 本体挖的"坑"，插件填内容     | `contributes.commands`、`contributes.views`        |
| **编程接口（API）**  | 衔接面 ③。本体↔插件的调用通道。`import vscode from 'vscode'`——插件能调的函数全在这里，不是 REST/gRPC，是代码层的 namespace | "你能碰什么由我说了算"       | `vscode.window`、`vscode.workspace`                |

> 三个衔接面各自的方向：Manifest = 插件告诉本体（单向）、扩展点 = 本体定义·插件填充（本体→插件）、编程接口 = 插件调本体（单向）。Manifest 太弱→本体必须运行代码才知道插件的能力；扩展点太乱→插件到处打洞；编程接口太宽→插件能绕过本体直接碰底层。

三个核心判断：

- **本体越稳定，后续产品启动越快。**（VSCode 十年不变的核心抽象，承载 20000+ 插件生态）
- **插槽越清晰，后续扩展越少侵入。**（Eclipse 每个扩展点有明确 Schema 和版本兼容承诺）
- **模组体越可复用，产品失败后的损失越小。**（推荐策略、风控、内容处理等模块可跨产品复用）

#### Q1 · 边界 —— 本体和插件怎么分？

**为什么重要**：这是第一个也是影响最深的设计决策——Core 放什么、插件放什么，决定了后续的扩展点粒度、隔离成本和生态自由度。Core 放多了→插件化形同虚设；Core 放少了→插件接入门槛高、框架脆弱。本质是**稳定性与灵活性的权衡**。

**定义**：决定哪些能力留在 Core 里长期不动、哪些作为插件/模块可变可换。

**调研情况**：

| #   | 分法                      | 解释                                                                                                                                                                     | 总结                                            | 代表                |
| --- | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------- | ------------------- |
| ①   | **瘦核心 + 厚插件**       | Core 只是一个"壳"——窗口管理、命令路由、插件生命周期调度。语言支持、主题、调试器全是插件。VSCode 启动时只扫 manifest 不加载代码，用户打开 Python 文件才激活 Python 插件。 | Core 是调度层，不懂业务，能力全部外置           | VSCode、Grafana     |
| ②   | **厚核心 + 薄插件**       | Core 本身是完整系统——有 DB、管理后台、页面渲染。插件不提供新能力，只在 Core 预留的 Hook 点注入回调。WordPress 预埋了 2000+ 个这样的点（`save_post`、`the_content`）。    | Core 是业务底座，插件只是在关键节点"顺便做点事" | WordPress、Drupal   |
| ③   | **全插件化**              | 没有 Core 和插件的区分。Eclipse 启动时连 OSGi 运行时本身也是一个 Bundle。任何插件可以定义自己的扩展点让其他插件扩展——插件之间形成网状依赖。                              | 平台即插件集合，连"核心"也是插件                | Eclipse、JupyterLab |
| ④   | **统一底座 + 子系统插件** | 底座只做横向共性（SSO、组织权限、工作台入口、消息通知、搜索索引、审计日志），不碰业务。各业务系统作为插件接入，提供 UI 入口 + 数据 API + 业务动作。                      | 底座管共性，插件提供业务能力，编排层跨插件调用  | 飞书、钉钉、企微    |

**选型信号**：

- 能力种类 > 20 且还在增长 → ①
- 核心流程稳定、只需注入点 → ②
- 平台自身也需要持续模块化演进 → ③
- 内部多个独立系统需要统一入口和跨系统调用 → ④

#### Q2 · 扩展点 —— 插件能接在哪些位置？

**为什么重要**：扩展点是插件和 Core 之间的**耦合面**。太松→插件无所不能但互相踩踏；太紧→插件只能做 Core 预设的事。好的扩展点设计让 20000 个插件能共存而不互相干扰。本质是**自由度与约束的平衡**。

**定义**：Core 暴露什么"插槽"让插件接入，以及插件通过什么机制与 Core 耦合。

**调研情况**（按结构化程度从强到弱排列）：

| #   | 机制                                    | 解释                                                                                                                                                                                                                                                                                                                  | 总结                                                                                                                                           | 约束力 | 代表              |
| --- | --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | ------ | ----------------- |
| ①   | **预定义插槽**                          | 类似乐高底板——底板预留凸点，积木只能卡进凸点。VSCode 预定义了 70+ 种插槽类型（命令、菜单、视图…），插件在 `package.json` 的 `contributes` 里声明往哪个插槽放什么。Core 启动只扫 manifest 写入 Registry，插件代码不运行。                                                                                              | 框架定位置，插件填内容，不运行代码就能知道插件有什么。                                                                                         | ★★★★★  | VSCode、Eclipse   |
| ②   | **执行流 Hook**                         | 类似马路边的摊位点——Core 沿执行流程摆好了摊位（Hook 点），插件知道摊位名就能摆摊。WordPress 预埋了 2000+ 个 `do_action('save_post')` 这样的点，插件 `add_action('save_post', callback)` 挂上去。Action 型只做事不返回值（"顺便发个通知"），Filter 型收数据改了再还回去（"帮改一下标题"）。                            | Core 沿路埋点，插件在感兴趣的点注册回调，门槛最低。                                                                                            | ★★★☆☆  | WordPress、Drupal |
| ③   | **接口 + 多实现**                       | 类似充电协议——手机只认"Type-C 协议"接口，充电头可以是小米/苹果/华为，插哪个用哪个，但一次只用一个。Core 定义 `PaymentProvider` 接口，支付宝、微信各实现一个，运行时按配置名选一个执行。                                                                                                                               | Core 定接口，插件给实现，多选一。适合支付渠道、推荐策略等可替换场景。                                                                          | ★★★★☆  | Java SPI、Grafana |
| ④   | **服务注册/发现**<br>（Token-based DI） | 类似插座标准——Token 是 USB-C 标准本身，Provider 是墙上的插座，Consumer 是电器。电器和插座都只认 USB-C，不绑定品牌。JupyterLab 的做法是 Token 从独立 npm 包导出——插件 A 声明 `provides: IToolbar`，插件 B 声明 `requires: [IToolbar]`，框架拓扑排序后先激活 A 再激活 B。换掉 A 的实现不需改 B。                        | Provider 和 Consumer 只认同一个 Token 对象，互不认识对方源码——通过独立包解耦。注意"Token-based DI"是 JupyterLab 自己的叫法，并非业界通用模式名 | ★★★★☆  | JupyterLab        |
| ⑤   | **声明式数据定义**                      | 类似 Excel 填表——不改程序逻辑，只填一张表就能加新内容。Skyrim 里 NPC、物品、法术都是一条 Record。Mod 作者写一个 `.esp` 文件只放"我要改的 Record"，游戏启动时按加载顺序合并——后加载的覆盖前加载的同 ID 字段，原版文件永远不动。**不需要写一行代码**。                                                                  | 扩展 = 填数据文件，引擎自动合并。适合让非程序员参与内容创作。                                                                                  | ★★☆☆☆  | Skyrim、RimWorld  |
| ⑥   | **运行时方法拦截**                      | 类似外卖中间人——你点餐→中间人记偏好→帮你下单→商家出餐→中间人先看→再递给你，全程中间人对你和商家透明。Harmony 分三级：Prefix 在方法执行前拦截（return false 可完全跳过原方法），Postfix 在方法执行后拦截（可读取修改返回值），Transpiler 直接修改方法本身的 IL 指令——强度最高也最危险，多个 Mod 同时改同一方法就冲突。 | 不改源码但拦截方法调用，是"最后手段"——只有 Core 没开扩展点又必须改行为时才用。                                                                 | ★☆☆☆☆  | Harmony、Mixin    |

#### Q3 · 声明 —— 宿主怎么知道有哪些插件？

**为什么重要**：Declaration is discovery——声明机制决定了 Core 在插件代码运行之前能知道多少信息。声明越结构化，Core 越能在启动时做全局决策（依赖解析、顺序编排、冲突检测）；声明越弱（代码即声明），Core 必须运行插件代码才知道它要做什么。本质是**静态可分析性 vs 接入门槛**的权衡。

**定义**：插件通过什么方式告诉宿主"我存在、我能做什么、我需要什么权限"。

**调研情况**：

| #   | 声明方式        | 解释                                                                                                                                                                                                                                                                                                                      | 总结                                     | 代表                       |
| --- | --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- | -------------------------- |
| ①   | **声明式 JSON** | 类似酒店入住——你填一张登记表放前台，前台扫一眼就知道你是谁、住几天、要不要早餐，不需要先把你带进房间。`plugin.json` 就是这张表，Core 启动时扫目录→`JSON.parse`→写入 Registry，全程不 `require()` 插件代码。JSON 解析是纯文本操作，扫 100 个插件也不慢。                                                                   | 不用进房间就知道你是谁，现代项目首选     | VSCode、Chrome、Grafana    |
| ②   | **声明式 XML**  | 类似签证申请表——格式严格，少填一栏直接退回，但能防造假。Eclipse 的 `plugin.xml` 有 XSD Schema 约束，少一个必填字段直接报错。Eclipse Runtime 解析 XML 后写入 Registry——**此时不创建 ClassLoader**，插件类只在被用户触发时才实例化。                                                                                        | 校验最严，Java 企业生态的历史选择        | Eclipse、IntelliJ          |
| ③   | **声明式 YAML** | 类似点菜——你跟服务员说"我要一份宫保鸡丁"，而不是"帮我切鸡肉、炒花生、调酱汁"。你声明的是期望结果，怎么做是厨房的事。`kubectl apply -f plugin.yaml` 后 K8s 把期望状态写入 etcd，Controller 发现"实际状态 ≠ 期望状态"就自动调谐——不需要你写"怎么装"的步骤。                                                                 | 声明期望结果，平台负责达成               | K8s、后端配置驱动项目      |
| ④   | **代码注册**    | 类似街头摆摊——不在管理处登记，直接找空位摆上就算开业了。WordPress 扫插件目录→`include` PHP 文件→include 时 `add_action('init', callback)` 立即执行，回调直接写进全局 Hook 表。没有中间层——**代码执行就是注册**。IntelliJ 则相反：扫 classpath→反射找 `@Plugin` 注解→写入 Registry，是结构化扫描，只是不需要独立声明文件。 | 门槛最低，但必须执行代码才知道注册了什么 | WordPress、IntelliJ        |
| ⑤   | **目录约定**    | 类似图书馆书架——书名、作者、分类不写在一张总表里，而是靠书在哪个架子、哪一层来约定。RimWorld 的 Mod 就是一个文件夹：`About/About.xml` 是封面、`Defs/` 是正文、`Assemblies/` 是附录。引擎扫 `Mods/` 目录，每个子文件夹就是一本"书"——**文件夹结构本身就是声明**，Mod 作者不需要填任何总表。                                 | 约定即声明，会建文件夹就能做 Mod         | RimWorld、Factorio、Skyrim |

**选型信号**：

- 新项目 → ①，搭配 JSON Schema 做校验
- K8s 生态或配置驱动 → ③
- 需要极低接入门槛、插件数 < 20 → ④
- 需要非程序员能手动创建插件 → ⑤

#### Q4 · 加载 —— 插件代码什么时候运行？

**为什么重要**：加载时机直接影响三个关键指标——**启动性能**（用户打开 App 要等多久）、**内存占用**（所有插件同时活着还是按需存活）、**安全**（代码是随包分发还是运行时动态下载）。本质是**即时可用性 vs 资源效率 vs 安全可控**的三角权衡。

**定义**：宿主什么时候把插件代码加载进内存并执行。

**调研情况**：

| #   | 策略                      | 解释                                                                                                                                                                                                                                                                                       | 总结                                         | 代表            |
| --- | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------- | --------------- |
| ①   | **声明式懒加载**          | 类似酒店叫醒服务——你跟前台说"我要赶早上6点的飞机"，前台记下了但不叫醒你。只有到6点才打电话。VSCode 的 `activationEvents: ["onCommand:x"]` 就是这个"叫醒条件"——Core 启动只读 manifest 记下条件，不加载代码，事件触发才 `require(main)`→`activate(context)`。20000+ 插件同时装也不拖慢启动。 | 启动快，谁用谁加载，不用不加载               | VSCode、Eclipse |
| ②   | **事件驱动即用即走**      | 类似外卖骑手——有订单才上线，送完就下线，不占内存。Chrome 的 Service Worker 不常驻：事件来→浏览器启动 SW→处理→空闲 30s→浏览器杀掉 SW。**不能依赖内存全局变量**——状态必须写 `chrome.storage`。数百个扩展的后台可能同时只有一个 SW 活着。                                                     | 资源利用率极高，但编程模型受限（随时被杀死） | Chrome MV3      |
| ③   | **分阶段加载**            | 类似盖房子——先打地基（Settings），再搭框架（Data，所有施工队一起上），最后装修（Runtime，各干各的）。Factorio 把初始化拆为三个阶段，Data Stage 所有 Mod 共享 Lua 状态互相协作构建数据，Runtime Stage 切回独立沙箱各自运行。**同一组 Mod 在不同阶段有不同的隔离规则**。                     | 复杂顺序可控，适合需要"先协作再独立"的场景   | Factorio        |
| ④   | **启动全量加载**          | 类似开学第一天全班点名——所有人到齐才开始上课。WordPress 进程启动→扫 `plugins/` 目录→逐个 `include` 文件→执行 `add_action()` 注册。简单直接，但插件多→启动慢、内存高。50 个插件的站点比 5 个的慢 3-5 倍。                                                                                   | 最简单直接，但只适合插件数 < 10 的场景       | WordPress       |
| ⑤   | **构建时注入 + 配置控制** | 类似预装软件——买手机时 App 已经在里面了，只是你可能看不到图标。模块代码随 App 编译进 APK/IPA，启动时只注册元数据不执行逻辑，远程配置中心控制入口可见/不可见。**代码从未被动态加载**——一直在包里，只是用户能不能看到入口。                                                                  | 最安全，不违反应用商店审核，能力变更需发版   | 移动端          |

**选型信号**：

- 插件数 > 50 → 必须用①或②
- 不可动态下发代码、需通过应用商店审核 → ⑤
- 所有 Mod 需要协作构建一个共享的数据世界再各自独立运行 → ③
- 插件数 < 10 且性能不敏感 → ④ 也够用

#### Q5 · 隔离 —— 一个插件崩了，谁受影响？

**为什么重要**：隔离决定了故障传播半径——插件 A OOM 会不会把 Core 和插件 B 一起带崩。隔离越强成本越高（进程开销/RPC 延迟/开发复杂度）。没有第三方插件的内部产品不需要重型隔离；对外开放的插件生态（插件可接触用户数据）必须有。本质是**安全 vs 成本**的权衡。

**定义**：插件和 Core 之间、插件和插件之间故障传播半径有多大。

**调研情况**（按隔离强度从低到高）：

| #   | 隔离方式             | 解释                                                                                                                                                                                                                           | 总结                                         | 强度  | 代表         |
| --- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------- | ----- | ------------ |
| ①   | **同进程无隔离**     | 类似合租一个房间——所有人共享一张桌子、一个插座。`add_action()` 直接把函数指针写进 Core 的全局 Hook 数组，调用就是直接 call，零开销也零保护。一个人打翻水杯，所有人的文件都湿了。                                               | 一个插件崩→全部崩。只有内部可信环境能用      | ★☆☆☆☆ | WordPress    |
| ②   | **ClassLoader 边界** | 类似同一栋楼的不同房间——每家有自己的门牌号和信箱（独立 ClassLoader，同名类互不可见），但共享水电总管线（线程/CPU/内存仍共享）。A 家的 `com.Util` 和 B 家的同名类各用各的，但楼上的水管爆了楼下也遭殃。                         | 类冲突可隔离，内存/CPU 故障仍扩散            | ★★☆☆☆ | Eclipse OSGi |
| ③   | **分阶段差异化**     | 类似盖房子——施工阶段（Data Stage）所有工队共享场地、自由协作、可以互相改图纸来协同建造；入住后（Runtime Stage）每家关门独立生活，必须敲门（`remote.call`）才能串门。同一栋楼，不同阶段规则完全不同。                           | 协作阶段开放，稳定阶段锁死——同一系统不同规则 | ★★☆☆☆ | Factorio     |
| ④   | **独立进程**         | 类似住在不同楼栋——A 楼着火不会烧到 B 楼。插件调 `vscode.window.xxx`→API Facade→序列化→RPC 管道→主进程执行→结果返回。插件进程崩了→主窗口弹"插件 X 挂了"，文件不丢。串门需要走街道（RPC），但换来的是安全。                      | 插件崩不影响 Core，适合有第三方插件的场景    | ★★★★☆ | VSCode       |
| ⑤   | **多上下文沙箱**     | 类似银行网点的三个角色——柜员（Service Worker）能碰金库但看不到客户；大堂经理（Content Script）能跟客户聊、看大厅里发生什么，但碰不到金库；你（Popup）只能在填单台的小隔间办自己的事。三者只能通过对讲机（`postMessage`）沟通。 | 每个上下文只拿最小能力，最安全，适合开放生态 | ★★★★★ | Chrome       |

**选型信号**：

- 插件都可信（内部团队开发）→ ① 或 ②
- 有第三方插件 → 至少④
- 对外开放生态、插件可接触用户数据 → ⑤

---

#### Q6 · 权限 —— 插件能访问什么数据和能力？

**为什么重要**：权限模型是一个插件平台的**信任基础设施**。如果插件默认就能读用户数据、发网络请求、写文件系统——那一个恶意插件就能造成灾难。反过来，如果权限设计过于复杂——门槛高到没人愿意开发插件。本质是**安全 vs 易用**的权衡，走的是**默认姿态**（进门时什么都不能做 vs 进门后什么都能做）到**运行时强制**（声明的权限是否真正检查）再到**可审计**（出问题能否追溯）的三层防御。

**定义**：插件默认什么都不能做（最小权限），还是默认什么都能做（信任底座），以及运行时如何强制执行。

**调研情况**（按默认姿态从紧到松排列）：

| #   | 模型                    | 解释                                                                                                                                                                                                                                                                                            | 总结                                           | 代表         |
| --- | ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------- | ------------ |
| ①   | **权限声明+安装时告知** | 类似入住酒店——你跟前台说"我要 WiFi、早餐、停车位"，前台给你对应的卡/券，没说的服务就没有。插件在 `manifest.json` 的 `permissions` 里逐项声明"我要读 storage""我要访问 example.com"，安装时弹出权限列表，用户同意才能装。运行时每次调 API 都检查 permissions 数组——没声明的 API 返回 undefined。 | 默认什么都不能做，声明一项给一项               | Chrome       |
| ②   | **API 收敛**            | 类似公司门禁卡——你刷卡只能进大厅和你的楼层，其他楼层刷卡无效。不是每扇门都问你"要进吗"，而是你的卡本来就只能进有限区域。插件 `import vscode from 'vscode'`——这就是它能访问的全部，`vscode` 命名空间不暴露文件系统、DOM、网络请求。                                                              | 不逐项声明，一个封闭的编程接口就是全部能力边界 | VSCode       |
| ③   | **默认拒绝+用户解锁**   | 类似 iPhone 的隐私弹窗——"允许微信访问你的相册吗？"默认不给，App 弹窗请求，你决定给不给。安装后 Restricted Mode 默认开启——**任何**社区插件禁止执行，用户必须在设置里显式关掉安全模式并确认风险。插件更新后要重新授权："上次你信的是 v3.2，现在是 v3.3，还信吗？"                                 | 比①更激进——直接不让跑，用户主动解锁才行        | Obsidian     |
| ④   | **签名分级信任**        | 类似三种证件——官方发的身份证（官方签名）直接进站；社区开的介绍信（社区签名）提交审核后放行；啥也没有（无签名）只能开发模式手动加载。信任不来自插件自己说了什么，来自谁给它签的名。                                                                                                              | 信任不来自声明，来自签名等级                   | Grafana      |
| ⑤   | **审核+沙箱**           | 类似机场安检+候机厅——先过安检机（代码审核），过了也只能在候机厅指定区域活动（独立 JS 引擎沙箱），不能去跑道、不能进驾驶舱、不能随意出航站楼（网络受域名白名单限制）。最严格的三重防线。                                                                                                         | 最重——审核+沙箱+域名白名单三重防线             | 微信小程序   |
| ⑥   | **信任底座+审计**       | 类似公司内部——员工默认可以进办公区用打印机、进茶水间，但每张门禁卡都有刷卡记录。`plugin.json` 声明权限 scope（L1只读→L5敏感），OAuth token scope 校验 + 全量审计日志。信任是默认的，但一切可追溯——出问题从审计日志还原"谁在什么时候调了什么"。                                                  | 默认信任，事后审计兜底——适合企业内部           | 内部办公平台 |

**选型信号**：

- 面向第三方、不可信 → ①+⑤
- 半开放、生态可控 → ②+④
- 企业内部、环境可控 → ⑥
- 普通用户为主、安全第一 → ③

---

#### Q7 · 通信 —— 插件与宿主、插件之间怎么对话？

**为什么重要**：通信是 Q1-Q6 全部选择的**收束点**——Q5 选了隔离方式就直接决定了通信的物理方式（同进程→直接调/跨进程→RPC/跨沙箱→消息传递）；Q2 选了扩展点机制就决定了通信的逻辑模式（Hook→隐式广播/服务注册→隐式注入/插件间→显式声明）。通信设计不好→要么插件间耦合成一团乱麻、要么出问题时追不到是谁调了谁。本质是**调用关系的可见性**。

**定义**：插件与 Core、插件与插件之间通过什么方式交换数据和调用能力，以及调用关系是否可追溯。

**调研情况**：

| #   | 方式                      | 解释                                                                                                                                                                                                                                                   | 总结                                                  | 代表                  |
| --- | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------- | --------------------- |
| ①   | **直接 API 调用**         | 类似在同一个房间说话——直接喊一声就行，零延迟。插件 `import vscode`→`vscode.window.showMessage('hello')`，看到的是 API Facade，看不到背后是直接调还是 RPC。本质是**插件→Core**的垂直调用。                                                              | 同进程，插件调 Core 暴露的 namespace                  | VSCode、WordPress     |
| ②   | **RPC**                   | 类似打电话——你看不到对方在干嘛，但说"帮我查一下库存"，对方查完告诉你结果。插件侧代码和①完全一样，但实际链路是 `调 Facade→JSON 序列化→管道/IPC→主进程反序列化→执行→结果序列化返回`。对插件透明——插件不知道自己在做 RPC。                                | 跨进程，有序列化开销，但换来了故障隔离                | VSCode                |
| ③   | **消息传递**              | 类似寄信——你把信塞进邮筒（`sendMessage`），邮局按地址送达，收信人拆开看。发完你就忙别的去了，不站在邮筒前等回复。Content Script、Service Worker、Popup 之间**不能共享变量、不能 import、只能发消息**。                                                 | 跨沙箱唯一通信方式，异步单向                          | Chrome                |
| ④   | **事件总线**              | 类似广播喇叭——大喇叭喊"张三你的快递到了"，谁在听谁去拿。Core `do_action('order_created', $order)` 发射事件→遍历全局 Hook 数组→按 priority 依次执行。喊的人不知道也不关心谁听到了——**隐式依赖**，出问题时无法 grep 追踪。                               | 解耦 1:N 通知，但依赖关系不可 grep 追踪               | WordPress Hook、Kafka |
| ⑤   | **显式 remote interface** | 类似对讲机——你调到仓库频道直接点名"仓库，查一下铁板库存"。Mod A 暴露 `remote.add_interface("warehouse", {get_stock=fn})`，Mod B `remote.call("warehouse", "get_stock", "iron-plate")`。**双方都知道对方存在**——grep `remote.call` 就能画出插件依赖图。 | 插件间通信的标准做法，显式优于隐式，grep 即可画依赖图 | Factorio              |

**选型信号**：

- Q5 选①或②（同进程）→ 用①（直接调用，如果 Core API 已收敛）或④（解耦 1:N）
- Q5 选④（独立进程）→ 必须用②（RPC）
- Q5 选⑤（多上下文沙箱）→ 必须用③（消息传递）
- 插件间通信 → ⑤（显式）优于④（隐式）。grep `remote.call` 就能画出插件依赖图

### 1.3 怎么选 —— 从三种典型起点出发

1.2 的菜单是"有什么可选"，本节是"怎么选"。不同产品做插件化的动机不同，起点也不同。这里从**三种最常见的起点**出发，给出 Q1→Q7 的推荐组合。如果你的情况横跨两种，按主导的那边取。

---

#### 起点一：我有一个产品，功能越来越多、越来越重

> 典型信号：首页塞满入口、菜单层级越来越深、一个简单需求改动要改几十个文件、新功能上线不敢快速迭代因为怕影响主流程。

这类产品的问题不是"缺少一个插件系统"，而是**本体和能力从来没被区分过**。

**核心思路**：先识别稳定的本体，再把可变能力外置。用户不应该感知"插件"这个词。

| Q         | 推荐                                                               | 为什么                                                                                           |
| --------- | ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------ |
| Q1 边界   | 识别本体的稳定部分（领域对象+主流程+状态规则），可变能力外置为模块 | 不是套框架模式——先回答"没有X产品还成立吗"。推荐/搜索/风控/AI 增强通常可从本体抽离                |
| Q2 扩展点 | ② 执行流 Hook + ③ 接口+多实现 + ⑤ 声明式数据定义                   | Hook 注入主流程节点（发布前/互动时/支付后）；接口+多实现切换策略；声明式数据定义让非程序员配模板 |
| Q3 声明   | ① JSON（`plugin.json`）                                            | 声明能力 scope、挂载位置、权限要求、配置项                                                       |
| Q4 加载   | ⑤ 构建时注入 + 配置控制                                            | 能力模块随 App 打包或服务端承载，远程配置控制灰度和可见性                                        |
| Q5 隔离   | ① 同进程无隔离（端上）+ 服务端承载重能力                           | 端上不做进程隔离；复杂计算（推荐/搜索/风控/内容处理）放服务端                                    |
| Q6 权限   | ② API 收敛 + ③ 默认拒绝+用户解锁                                   | 编程接口保护内部实现；用户数据访问必须授权                                                       |
| Q7 通信   | ① 直接 API 调用 + ④ 事件总线                                       | 同模块内直接调；跨模块用事件解耦                                                                 |

**落地路径**：

> 1. **画本体图**——列出领域对象、核心主流程、关键状态规则。问每个东西"没有它产品还成立吗"。
> 2. **抽能力模块**——把可变能力按平台能力（账号/支付/通知）、业务能力（推荐/搜索/内容处理）、增强能力（AI/分析/自动化）分层。
> 3. **定义扩展点**——只开 3-5 个核心插槽（主流程 Hook、策略替换位、首页/详情操作区入口），不贪多。
> 4. **建配置控制**——能力模块的开放通过远程配置中心控制，灰度→AB实验→全量→降级，异常时一键关闭入口不影响主流程。
> 5. **用户侧包装**——按用户任务（"更快发布""更安全使用"）组合能力入口，用户感知不到"插件"。

> 这个起点覆盖 ToC 产品（社交/内容/电商/婚恋/邮箱/工具…）和功能膨胀的自研工具。关键不是建一个插件系统，而是**把本体和能力拆开**。

#### 起点二：我有多个独立系统，需要统一入口和跨系统协作

> 典型信号：火星 im、pmgo、会议系统、a31、在线文档、kms 各自独立，用户需要在系统之间搬运信息、重复登录、手动推进流程。

这类问题的本质不是"做一个更大的 OA"，而是**让系统从入口升级为能力**——每个系统不仅要能被打开，还要能被平台搜索、调用、编排。

**核心思路**：先建统一底座承载横向共性，再让各系统暴露 API 和动作，最后由编排层跨系统调度。

| Q         | 推荐                                                       | 为什么                                                                        |
| --------- | ---------------------------------------------------------- | ----------------------------------------------------------------------------- |
| Q1 边界   | ④ 统一底座 + 子系统插件                                    | 底座管入口/身份/权限/消息/搜索/审计；各业务系统作为插件接入                   |
| Q2 扩展点 | ① 预定义插槽 + ② 执行流 Hook                               | 插槽定义工作台菜单、消息卡片、审批节点、搜索索引；Hook 覆盖流程中"顺便做点事" |
| Q3 声明   | ① JSON（`plugin.json`）+ 配置页                            | JSON 声明能力（API/动作/权限），配置页可视化管理                              |
| Q4 加载   | ① 声明式懒加载                                             | 用户进入工作台入口才激活对应子系统，不访问不加载                              |
| Q5 隔离   | ① 同进程无隔离（可信环境，通过编程接口调用，不直连数据库） | 企业内可控，不需要进程隔离                                                    |
| Q6 权限   | ⑥ 信任底座 + 审计                                          | OAuth scope 细粒度（L1只读→L5敏感操作）、回调签名校验、全量审计               |
| Q7 通信   | ① 直接 API 调用 + ④ 事件总线 + 编排层                      | 子系统间不直接互调，统一通过编排层（流程引擎/AI Agent）调度                   |

**落地路径**：

> 1. **建统一底座**——SSO、组织权限、统一工作台、统一消息/待办、统一搜索索引。
> 2. **补只读 API**——每个子系统先暴露查询 API，编排层能跨系统读数据。先做"项目进展总结""待办汇总"等纯读 PoC。
> 3. **封业务动作**——把高频操作（建任务、发通知、发起审批）封装为标准动作，注册到能力注册中心。
> 4. **编排层调度**——编排层（流程引擎/AI Agent）按用户目标跨插件调用能力。先只读，再受控写（所有写操作需用户确认 + 审计日志）。
> 5. **沉淀规范**——`plugin.json` 规范、插件 SDK、UI 插槽规范、版本管理、下线机制。

> 这个起点覆盖内部办公平台（OA/报表/流程/知识库…）。关键不是做一个更大的入口门户，而是**每个系统都要从"页面入口"升级为"能力提供方"**。

#### 起点三：我在做一个工具/平台，未来会有第三方或社区贡献插件

> 典型信号：你做的不是"一个产品"，而是"一个别人可以扩展的平台"——比如 IDE、数据可视化平台、知识管理工具、游戏、浏览器。

这类产品的插件化不是内部模块化的问题，而是**开放给别人写插件**的问题。关键决策点：插件作者是谁（内部/外部/完全开放），以及对应的安全级别。

| Q         | 推荐（按插件来源分级）                                                                                                   | 为什么                                                                         |
| --------- | ------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------ |
| Q1 边界   | ① 瘦核心 + 厚插件                                                                                                        | 只保留调度层，能力全部外置——VSCode、Grafana、Eclipse 都是这个模式              |
| Q2 扩展点 | ① 预定义插槽（主）+ ② 执行流 Hook（辅）                                                                                  | 预定义插槽是开放平台的标准做法——不定义清楚"哪里可以扩展"，第三方插件会到处打洞 |
| Q3 声明   | ① JSON（`manifest.json`）                                                                                                | 标准格式，每个插件作者都能看懂                                                 |
| Q4 加载   | ① 声明式懒加载                                                                                                           | 插件数量不可控时必须懒加载——VSCode 20000+ 插件全靠这个                         |
| Q5 隔离   | **内部团队** → ① 同进程；**可信第三方** → ④ 独立进程；**完全开放** → ⑤ 多上下文沙箱                                      | 隔离级别直接取决于插件作者是谁                                                 |
| Q6 权限   | **内部团队** → ⑥ 信任底座+审计；**可信第三方** → ② API收敛+④签名分级；**完全开放** → ① 权限声明+安装时告知 + ⑤ 审核+沙箱 | 安全模型也取决于插件来源——同一个人写的不需要重型权限                           |
| Q7 通信   | ② RPC + ① 直接 API 调用                                                                                                  | 靠 Q5 决定——独立进程→RPC，同进程→直接调用                                      |

**落地路径**：

> 1. **定插件来源和信任级别**——内部团队、可信第三方、还是完全开放？这决定 Q5 和 Q6 的选型。
> 2. **定义 Core 和首批扩展点**——梳理最需要被扩展的 5 个位置，定义为第一批 Contribution Points。
> 3. **设计 Manifest**——`contributes` 静态声明 + `activationEvents` 激活条件 + `engines` 版本兼容。
> 4. **选隔离方案并实现 Host**——内部→同进程，可信→独立进程，开放→多上下文沙箱。
> 5. **建分发和审核机制**——内部→配置管理，可信→签名+商店，开放→审核+签名+商店。

> 这个起点覆盖 PC 工具（VSCode/Eclipse 模式）、Web 平台（Chrome Extension 模式）、数据平台（Grafana 模式）、游戏 Mod（RimWorld/Skyrim 模式）。各端的硬约束（包体/启动/审核）在落地路径第 4 步才考虑，第 1-3 步是通用的。

---

#### 起点之外：各端落地时的额外约束

以上三个起点决定 Q1-Q7 的选型，但不同端有各自的硬约束：

| 端         | 额外约束                              | 影响哪些 Q                                                      |
| ---------- | ------------------------------------- | --------------------------------------------------------------- |
| **PC 端**  | 可接受进程隔离开销，启动性能敏感      | Q5（独立进程可行）、Q4（必须懒加载）                            |
| **Web 端** | 浏览器沙箱限制，无法做进程隔离        | Q5（只能多上下文沙箱）、Q7（只能消息传递）                      |
| **移动端** | 应用商店审核，包体/启动/内存/电量敏感 | Q4（必须构建时注入）、Q5（不能进程隔离）、Q6（必须签名+白名单） |
| **后端**   | 多服务协作，可独立部署，治理要求高    | Q2（优先接口+多实现+CRD）、Q5（独立进程+gRPC）                  |

## 二、附录 A：扩展点详解

> 对应 [1.3 Q2](#q2--扩展点----插件能接在哪些位置)「扩展点——插件能接在哪些位置」。做法菜单里列出了 6 种机制的基本形态和选型信号，本章是每种机制的代表产品拆解与代码级细节，需要深入某一种时按需查阅。

### 2.1 扩展点的本质

扩展点 = **Core 预定义的、插件可以贡献能力的固定位置**。

核心原则：

- 扩展点必须先定义再使用，插件不能随意在任何位置注入
- 扩展点要少而稳，而非多而杂
- 每个扩展点有明确的输入/输出/生命周期约定
- 扩展点本身也应该可以版本演进

### 2.2 各端的扩展点类型

#### PC 端（VSCode 模式）

VSCode 通过 Contribution Points 定义了 70+ 种扩展点，常见类型：

| 扩展点类型      | 说明                         | 典型用途                   |
| --------------- | ---------------------------- | -------------------------- |
| `commands`      | 注册可执行命令               | 导出、搜索、格式化         |
| `menus`         | 向菜单/右键菜单增加入口      | 编辑器右键、资源管理器右键 |
| `views`         | 增加侧边栏面板               | 文件树、大纲、调试面板     |
| `languages`     | 语言支持（高亮、补全、诊断） | TypeScript、Python 支持    |
| `debuggers`     | 调试器接入                   | Node.js、Python 调试       |
| `themes`        | 主题和图标主题               | 暗色/亮色主题              |
| `configuration` | 配置项定义                   | 编辑器设置、插件专属配置   |

**设计要点**：

- 静态声明（`contributes`）+ 动态注册（`activate()` 中注册 handler），两阶段分离
- `contributes.commands` 让命令在启动时即可见，但 handler 在插件激活后才绑定
- 插件的代码在 `activationEvents` 触发前完全不运行

#### Web 端（Chrome Extension 模式）

Chrome 的扩展点和权限强绑定：

| 扩展点                      | 用途                  | 所需权限                |
| --------------------------- | --------------------- | ----------------------- |
| `action`                    | 工具栏按钮和 popup    | 无需额外声明            |
| `content_scripts`           | 向匹配网页注入 JS/CSS | `host_permissions`      |
| `background.service_worker` | 后台事件处理          | 按需声明具体 API 权限   |
| `contextMenus`              | 右键菜单              | `contextMenus`          |
| `declarativeNetRequest`     | 声明式网络规则        | `declarativeNetRequest` |
| `commands`                  | 快捷键                | 无需额外声明            |

**核心特点**：能做什么取决于 manifest 声明了什么权限——扩展点即权限边界。

#### 后端（多种模式共存）

| 扩展点类型        | 机制                                                               | 代表                               |
| ----------------- | ------------------------------------------------------------------ | ---------------------------------- |
| Hook/Filter       | 观察者模式，在特定时刻执行代码                                     | WordPress(do_action/apply_filters) |
| Strategy/Plugin   | 可替换实现，多选一                                                 | Drupal Plugin API, Java SPI        |
| Middleware Chain  | 链式处理                                                           | Express/Koa 中间件, Nginx 模块     |
| CRD + Operator    | 声明式资源扩展                                                     | Kubernetes                         |
| Service Interface | 插件声明 provides/requires，框架按拓扑排序注入，Token 从独立包导出 | JupyterLab                         |
| LSP Protocol      | 协议级扩展                                                         | Language Server Protocol           |

**选型指南**（来自 Drupal 的启发）：

```
需要做什么？
├─ 在某个时刻「顺便做点事」？ → Hook
├─ 提供「多种可选方案」让调用方选择？ → Strategy/Plugin
├─ 提供「全局唯一的工具/能力」？ → Service/DI
├─ 「解耦发布-订阅」通信？ → Event
└─ 声明式资源管理？ → CRD + Operator
```

#### 移动端

移动端 UI 空间有限，插槽要少而稳：

| 插槽             | 说明                     | 示例                       |
| ---------------- | ------------------------ | -------------------------- |
| 首页卡片插槽     | 首页信息流或功能卡片位置 | 新手引导、活动卡、快捷服务 |
| 个人中心入口插槽 | 个人中心里的功能入口     | 会员权益、工具入口         |
| 详情页操作插槽   | 详情页里的操作按钮区     | 分享、收藏、辅助操作       |
| 发布页工具插槽   | 用户创作或发布时的工具区 | 模板、素材、增强工具       |
| 消息通知插槽     | 通知、待办、提醒入口     | 系统提醒、任务提醒         |

原则：移动端插槽 5-6 个就够，多了会导致页面拥挤、体验不一致。

#### 内部办公

内部办公的扩展点是**业务能力接入点**：

| 扩展点       | 说明                       |
| ------------ | -------------------------- |
| 工作台菜单   | 子系统入口挂载位置         |
| 消息卡片     | 通知/提醒/待办的卡片模板   |
| 机器人回调   | 对话交互的 Webhook         |
| 审批流节点   | 审批链路中的自定义处理节点 |
| 文档嵌入区块 | 在文档中嵌入业务数据       |
| 搜索索引     | 跨系统统一检索的数据源     |

#### ToC 产品

ToC 产品的扩展点不一定是技术上的"插槽"，更多是**产品主流程中允许能力模块接入的节点**：

| 扩展点类型          | 说明                           | 示例（以社交/内容产品为例）                                                       |
| ------------------- | ------------------------------ | --------------------------------------------------------------------------------- |
| **主流程节点**      | 核心路径上可插入增强能力的步骤 | 发布前（内容增强/审核）、发布后（推荐策略/分发渠道）、互动时（评论辅助/敏感过滤） |
| **功能入口**        | 用户可发现和使用能力的入口     | 发布页工具栏、详情页操作区、个人中心、首页卡片                                    |
| **内容处理 Hook**   | 内容生命周期各阶段的处理点     | 上传时（压缩/转码/审核）、展示时（渲染/翻译/摘要）、分发时（推荐/去重/风控）      |
| **互动与交易 Hook** | 用户行为链路中的切入点         | 关注后（推荐相似用户）、下单前（风控/优惠）、支付后（会员权益/积分）              |
| **通知与触达点**    | 可插入用户触达消息的位置       | 推送通道、站内信、邮件、短信                                                      |
| **数据与实验点**    | 可接入数据采集和实验的位置     | 埋点节点、AB 实验分流、用户画像标签                                               |

**ToC 扩展点的特殊性**：

- 扩展点不是让"第三方开发者"用的，首先是让自己团队的**能力模块**用的
- 扩展点不应该让用户感知到——用户看到的是"发布时有个美化按钮"，不是"发布页挂载了一个内容增强插件"
- 扩展点要和运营配置绑定：同一个扩展点，不同人群、不同版本、不同地区挂载不同能力
- ToC 产品的大部分扩展点不在端上，而在**服务端**——搜索、推荐、风控、内容处理、数据分析等复杂能力尽量服务端承载

**ToC 产品案例速查**：

| 产品类型 | 产品本体（不变）                    | 能力模块（可变/可复用）                            | 扩展点                                             |
| -------- | ----------------------------------- | -------------------------------------------------- | -------------------------------------------------- |
| 婚恋 App | 账号/资料/认证/匹配/聊天/风控       | 资料完善、匹配策略、聊天辅助、安全提醒、会员权益   | 资料页入口、匹配页卡片、聊天页建议区、个人中心     |
| 邮箱 App | 账号/收发邮件/联系人/附件/搜索/安全 | 邮件分类、重要提醒、摘要、附件处理、日程识别       | 首页摘要卡、邮件详情操作区、通知提醒、搜索结果扩展 |
| 内容社区 | 账号/Feed/发布/评论/推荐/风控       | 内容增强、翻译本地化、话题推荐、评论管理、增长分析 | 发布页工具栏、内容详情操作区、创作者中心卡片       |

Chrome 的扩展点和权限强绑定：

来自 Eclipse 的 9 条设计铁律（Gamma & Beck）中，最关键的 5 条：

| 法则         | 含义                                                  |
| ------------ | ----------------------------------------------------- |
| **显式扩展** | 明确声明哪里可以被扩展，不允许随意注入                |
| **多样性**   | 每个扩展点必须接受多个扩展（1:N，不是一个萝卜一个坑） |
| **公平竞争** | 定义扩展点的模块自己也通过同一套机制使用它            |
| **好篱笆**   | 如果扩展出错，卸载它，不再调用                        |
| **懒加载**   | 扩展只在实际需要时才被加载                            |

补充两条实践原则：

- **插槽先收敛再开放**：先开 3-5 个核心插槽验证闭环，再逐步增加
- **每个插槽都要有 schema 约束**：输入什么、输出什么、何时触发、如何降级

## 三、附录 B：生命周期详解

> 对应 [1.3 Q3](#q3--声明----宿主怎么知道有哪些插件)「声明——宿主怎么知道有哪些插件」和 [Q4](#q4--加载----插件代码什么时候运行)「加载——插件代码什么时候运行」。本章展开 VSCode 的两阶段模型、Chrome 的 Service Worker、Factorio 的分阶段加载等代表案例。

### 3.1 生命周期要解决的问题

```
插件的完整生命周期：
安装 → 注册(静态声明) → 激活(代码运行) → 运行 → 停用 → 升级 → 卸载
```

核心矛盾：**插件越多，全量加载越慢；但不加载，用户用的时候就需要等待。**

解决方案：**声明式静态注册 + 按需动态激活**。

### 3.2 VSCode 的两阶段模型

这是目前最成熟的懒加载模型：

```
阶段一：启动扫描（不运行插件代码）
  1. VS Code 启动
  2. 扫描已安装插件目录
  3. 读取 package.json
  4. 解析 contributes（命令、菜单、视图等静态声明）
  5. 注册到各个 Registry
  6. 插件 JS 代码完全没有运行

阶段二：按需激活（用户触发时才加载代码）
  1. 用户触发 activationEvents 中声明的条件
  2. Extension Host 加载插件入口 main
  3. 调用 activate(context)
  4. 插件注册 handler/provider/listener
  5. 执行真正的业务逻辑
```

**激活事件类型**：
| 事件 | 触发时机 |
|------|---------|
| `onCommand:x` | 执行某命令时 |
| `onLanguage:x` | 打开某语言文件时 |
| `onView:x` | 视图展开时 |
| `onStartupFinished` | 启动完成后 |
| `*` | 始终激活（不推荐） |

### 3.3 Chrome / Web 端的事件驱动 Service Worker

Manifest V3 的核心变革：**从持久后台页转向事件驱动 Service Worker**。

```
旧模式 (MV2)：background page 常驻内存，持续运行
新模式 (MV3)：Service Worker 事件来了启动，空闲30秒即终止

影响：
- 不能再依赖全局变量保存状态 → 必须用 chrome.storage
- 事件监听必须在脚本顶层注册（不能在异步回调里注册）
- 内存占用大幅降低
```

### 3.4 Factorio 的三阶段生命周期

分阶段设计是处理复杂初始化顺序的典范：

```
Phase 1: Settings Stage
  - 每个 Mod 独立的 Lua 状态
  - 产出：用户配置的 Mod 设置

Phase 2: Data Stage（关键设计：所有 Mod 共享 Lua 状态）
  - data.lua → data-updates.lua → data-final-fixes.lua
  - Mod 之间可以互相访问原型数据和函数
  - 产出：data.raw（不可变原型表）
  - 设计意图：允许 Mod 间自由协作构建数据

Phase 3: Runtime Stage
  - 每个 Mod 沙箱隔离
  - 必须通过 remote.call / remote.add_interface 显式跨 Mod 通信
```

**启发**：复杂初始化可以分阶段，不同阶段采用不同的隔离策略。

### 3.5 ToC 产品的生命周期特点

ToC 产品的"插件"生命周期和 PC/后端有本质区别——能力模块不一定是"运行时可加载的代码包"：

```
形态一：内置模块 + 配置控制
  构建时打包 → 启动时注册（不执行） → 用户触发场景时激活 → 运营配置控制可见性

形态二：服务端承载
  端上只有入口 → 用户触发 → 调用服务端能力 → 返回结果 → 端上展示
  能力代码完全在服务端，端上无加载成本

形态三：动态资源包
  端上内置核心模块 → 模板/素材/模型/规则按需下载 → 资源版本管理 → 异常时回退内置版本
```

**ToC 生命周期核心原则**：

- **构建期 > 启动期 > 运行时**：能构建时确定就别启动时判断，能启动时判断就别运行时加载
- **配置驱动的激活**：能力的"激活"不等于"加载代码"，更常见的是"配置开启入口"
- **服务端承载优先**：复杂计算、频繁变化的策略、大数据处理全部服务端化，端上只留轻入口
- **降级内置**：动态资源加载失败时回退到 App 内置版本，保证主流程不受影响

### 3.6 Eclipse 的懒加载注册表

```
启动 → 扫描 plugin.xml → 建立 Extension Registry → UI 可见扩展
→ 用户触发功能 → 按需调用 createExecutableExtension("class") → 插件代码首次运行
```

**关键**：Eclipse Registry 可以解析 `plugin.xml` 而**不创建 ClassLoader**。插件类只在 `createExecutableExtension()` 被调用时才加载。

### 3.7 生命周期设计 checklist

| 决策点           | 选项                | 适用场景                                 |
| ---------------- | ------------------- | ---------------------------------------- |
| **何时发现插件** | 启动时扫描          | 插件数量可控，目录固定                   |
|                  | 运行时注册          | 动态增减插件                             |
|                  | 构建时注入          | 移动端内置模块                           |
| **何时激活插件** | 声明式事件          | 插件多、性能敏感的客户端                 |
|                  | 启动即全量          | 插件少、可靠性要求高                     |
|                  | 用户手动触发        | 安全性要求高（Obsidian Restricted Mode） |
| **如何停用插件** | Disposable 清理     | 注册的所有资源必须可释放                 |
|                  | 进程/线程终止       | 进程隔离模式                             |
|                  | 状态序列化+恢复     | Service Worker 模式                      |
| **如何升级插件** | 替换文件+重载       | 客户端插件                               |
|                  | 新版本号+注册表切换 | 后端服务                                 |
|                  | 灰度+切换流量       | 后端/移动端                              |

## 四、附录 C：运行隔离详解

> 对应 [1.3 Q5](#q5--隔离----一个插件崩了谁受影响)「隔离——一个插件崩了，谁受影响」。本章展开 VSCode 的进程隔离、Chrome 的多上下文沙箱、Eclipse 的 ClassLoader 隔离、移动端的受限隔离四种方案的技术细节。

### 4.1 隔离光谱

```
隔离程度从低到高：

WordPress     ─ 同一进程，Hook 机制，无隔离 ★☆☆☆☆
Factorio      ─ Data Stage 共享，Runtime 沙箱 ★★☆☆☆
RimWorld      ─ 同一 CLR，Harmony 可互相影响 ★★☆☆☆
Obsidian      ─ 同一 Electron 进程，Restricted Mode ★★★☆☆
VSCode        ─ Extension Host 独立进程 ★★★★☆
Chrome        ─ Service Worker 独立 + Content Script 沙箱 ★★★★★
Eclipse       ─ OSGi ClassLoader 级别隔离 ★★★★★
```

### 4.2 隔离方案详解

#### 方案A：进程隔离（VSCode）

```
VSCode 主进程 (Electron 渲染进程)
│
├─ Plugin Manager：安装、启用、禁用、卸载
│
└─ Extension Host (独立 Node.js 进程)
      ├─ 插件 A
      ├─ 插件 B
      └─ 插件 C
```

**效果**：

- Extension Host 崩溃 → 编辑器窗口不受影响
- 插件不能直接访问 DOM 和 Node.js API，只能通过 `vscode` 命名空间
- 插件 CPU 占用可以被宿主进程监控和限制

**通信方式**：Extension Host ↔ 主进程 通过 RPC 协议通信，插件调用 `vscode.window.showInformationMessage()` 实际链路是 `插件代码 → API facade → Extension Host → RPC → 主线程 → UI`。

#### 方案B：多上下文沙箱隔离（Chrome）

Chrome 把扩展拆到多个运行上下文，每个上下文只拿最小能力：

| 运行环境           | 能访问什么                      | 不能访问什么                                   |
| ------------------ | ------------------------------- | ---------------------------------------------- |
| **Service Worker** | Chrome Extension API、事件      | DOM、页面 JS 变量                              |
| **Content Script** | 页面 DOM、部分扩展消息 API      | 其他扩展的变量、页面 JS 变量（isolated world） |
| **Popup Page**     | Popup DOM、Chrome Extension API | 网页 DOM                                       |
| **Web Page**       | 页面自己的 DOM 和 JS            | 扩展内部 API                                   |

**关键设计**：

- Content Script 运行在 "isolated world"——能看到页面 DOM，但 JS 变量对页面和其他扩展不可见
- 跨上下文通信只能通过 `chrome.runtime.sendMessage`
- Service Worker 事件驱动，空闲即终止——不假设自己长期存活

#### 方案C：ClassLoader 级隔离（Eclipse/Java）

```
每个插件 = 一个 OSGi Bundle
  - 独立的 ClassLoader
  - 声明导出哪些 package
  - 声明依赖哪些 bundle
  - 不能随意访问其他 bundle 的内部类
```

**效果**：同一 JVM 内实现了模块间强隔离，但不提供进程级故障隔离。

#### 方案D：移动端的受限隔离

移动端无法像桌面那样做进程隔离，但可以分层约束：

```
App 本体层
  └─ 统一授权：相册/定位/通讯录/通知等敏感权限统一管控
  └─ 模块不能绕过 App 本体直接访问敏感权限
  └─ 模块不能私自上传用户数据

受控模块层
  └─ 只通过 App 本体暴露的 Bridge API 调用原生能力
  └─ 模块调用全量埋点和异常监控
  └─ 写操作/支付操作必须有确认和风控
```

### 4.3 隔离设计原则

| 原则           | 说明                                             |
| -------------- | ------------------------------------------------ |
| **默认隔离**   | 插件不应默认获得宿主全部能力                     |
| **最小权限**   | 插件只拿到完成任务所需的最小能力集合             |
| **安全通信**   | 跨隔离边界通信通过显式的消息传递/RPC，不共享内存 |
| **故障不扩散** | 单个插件异常不应影响宿主和其他插件               |
| **可恢复**     | 插件崩溃后可以重新加载，状态不丢失               |

### 4.4 隔离与通信的配套

隔离不是目的，安全通信才是。不同隔离等级需要匹配不同的通信模式：

| 隔离等级               | 通信方式                | 延迟 | 复杂度 |
| ---------------------- | ----------------------- | ---- | ------ |
| 无隔离（同进程）       | 直接函数调用            | 极低 | 极低   |
| 线程隔离               | 消息队列/Channel        | 低   | 中     |
| ClassLoader 隔离       | 接口调用+反射           | 低   | 中     |
| 进程隔离               | RPC/gRPC/Pipe/stdio     | 中   | 高     |
| 沙箱隔离（不同上下文） | postMessage/sendMessage | 中   | 高     |
| 远程隔离               | HTTP/gRPC/WebSocket     | 高   | 最高   |

## 五、附录 D：安全模型详解

> 对应 [1.3 Q6](#q6--权限----插件能访问什么数据和能力)「权限——插件能访问什么数据和能力」。本章展开 6 种安全模型的技术实现与 checklist。

### 5.1 安全模型对比

| 模型            | 代表              | 核心思想                                               | 适用场景                     |
| --------------- | ----------------- | ------------------------------------------------------ | ---------------------------- |
| **权限声明式**  | Chrome            | Manifest 声明权限，安装时告知用户，运行时强制检查      | 开放生态，第三方不可信       |
| **API 收敛式**  | VSCode            | 只暴露稳定 API facade，不暴露内部实现                  | 半开放生态，插件可信但需约束 |
| **默认拒绝式**  | Obsidian          | Restricted Mode 默认开启，用户主动关闭才启用第三方插件 | 普通用户为主的产品           |
| **信任底座式**  | Eclipse、内部办公 | 假设插件可信，通过模块依赖和 API 边界约束              | 企业内可控生态               |
| **沙箱+审核式** | 微信小程序        | 平台审核 + 运行时沙箱 + 域名白名单                     | 超级 App 生态                |

### 5.2 安全设计的核心维度

#### 维度一：权限声明

Chrome 的模式最适合直接借鉴：

```
插件要访问文件系统 → 声明 fileSystem
插件要访问网络 → 声明 network
插件要访问用户数据 → 声明 userData
插件要修改页面 → 声明 pageInjection
插件要执行脚本 → 声明 scripting
```

对内平台也可以做权限分级：

```
L1: 只读公开数据
L2: 读取用户相关数据
L3: 写入业务数据
L4: 触发通知/审批
L5: 执行敏感操作（支付、导出、删除）
```

#### 维度二：运行沙箱

- PC 端：Extension Host 独立进程
- 浏览器端：Content Script isolated world + Service Worker
- 移动端：原生能力统一授权，模块不能绕过 App 本体
- 后端：独立进程 + 网络策略 + 资源配额

#### 维度三：签名与审核

Grafana 的签名分级模型值得参考：

```
签名等级：
- 官方签名：Grafana 团队开发，最高信任
- 社区签名：社区开发者，经过审核
- 无签名：本地开发/企业内部，只能开发者模式加载
```

#### 维度四：数据边界

| 控制点         | 做法                               |
| -------------- | ---------------------------------- |
| 插件能读什么   | 声明数据 scope，按需授权           |
| 插件能写什么   | 写操作必须有用户确认或审批         |
| 插件能外传什么 | 网络白名单，敏感数据脱敏           |
| 插件能存什么   | 隔离存储空间，不能访问其他插件数据 |

### 5.3 安全设计原则

```
1. 默认拒绝（Default Deny）
   - 插件默认无权限，必须显式声明和获取授权
   - 参考：Obsidian Restricted Mode、Chrome MV3 权限模型

2. 最小权限（Least Privilege）
   - 每个插件只获得完成任务所需的最小权限集合
   - 参考：Chrome 的 permissions 数组 + host_permissions

3. 纵深防御（Defense in Depth）
   - 多层安全：声明层 + 运行时检查 + 数据边界 + 审计
   - 参考：Chrome 的 Manifest 声明 + Content Script 隔离 + Service Worker 事件驱动

4. 安全默认 > 用户选择
   - 默认安全比让用户选择更重要
   - 参考：Obsidian Restricted Mode 默认开启

5. 可审计（Auditable）
   - 所有插件的权限使用、API 调用、数据访问都要可追踪
   - 对内系统尤其重要：谁在什么时候调用了什么能力
```

### 5.4 安全 checklist（按端）

**PC 端**：

- [ ] 插件代码运行在独立进程
- [ ] 不能直接访问文件系统（通过 API 中转）
- [ ] 不能直接访问 DOM
- [ ] 网络请求受控
- [ ] 插件签名校验

**移动端**：

- [ ] 敏感权限（相机/定位/通讯录）由 App 本体统一管控
- [ ] 模块不能私自上传用户数据
- [ ] 支付必须走统一支付风控链路
- [ ] 动态加载的代码必须签名校验

**后端**：

- [ ] 插件只能访问授权范围内的数据
- [ ] 插件调用有速率限制（rate limiting）
- [ ] 插件异常有熔断和降级
- [ ] mTLS 或 Token 认证
- [ ] 所有调用可审计

**内部办公**：

- [ ] OAuth scope 细粒度控制
- [ ] 回调 URL 签名校验
- [ ] 租户数据隔离
- [ ] 写操作需要用户确认
- [ ] 高风险操作设置审批

**ToC 产品**：

- [ ] 能力模块通过声明明确读写权限 scope
- [ ] 用户数据访问必须经过授权（OAuth scope / 用户主动授权）
- [ ] 写操作（支付、发布、删除）必须有用户确认 + 风控
- [ ] 敏感权限（相册/定位/通讯录）由 App 本体统一管控，模块不能绕过
- [ ] 动态资源/代码签名校验 + 传输加密
- [ ] 网络请求域名白名单 + 敏感数据脱敏后外传
- [ ] 配置回滚/降级机制：某个能力异常可一键关闭入口，不影响主流程

## 六、附录 E：实现方案速查

> 对应 [1.3 Q3](#q3--声明----宿主怎么知道有哪些插件)「声明」、[Q4](#q4--加载----插件代码什么时候运行)「加载」、[Q7](#q7--通信----插件与宿主插件之间怎么对话)「通信」，以及发现机制、版本兼容、分发渠道等横向话题。本章提供声明格式选型对比、加载策略对比、通信模式对比、分发机制等速查表。

### 6.1 声明机制对比

| 方案        | 格式                                         | 代表                    | 优点                              | 缺点                         |
| ----------- | -------------------------------------------- | ----------------------- | --------------------------------- | ---------------------------- |
| 声明式 JSON | `package.json`/`manifest.json`/`plugin.json` | VSCode, Chrome, Grafana | 结构清晰、机器可读、IDE 可提示    | 表达能力有限                 |
| 声明式 XML  | `plugin.xml`                                 | Eclipse, IntelliJ       | Schema 校验、历史悠久、工具链成熟 | 冗长、现代项目不偏好         |
| 声明式 YAML | `plugin.yaml`/CR yaml                        | K8s, 部分后端           | 人类可读性好                      | 缩进敏感                     |
| 代码注册    | `add_action()`/`add_filter()`                | WordPress               | 门槛极低、灵活                    | 无结构约束、难以静态分析     |
| 代码声明    | `@Plugin` 注解 / `JupyterFrontEndPlugin`     | IntelliJ, JupyterLab    | 类型安全                          | 需要编译、不读代码不知道能力 |

**推荐**：新项目优先选 JSON，配合 JSON Schema 做校验。JSON 是各语言通用、各类工具都能处理的格式。

### 6.2 发现机制对比

| 方案               | 时机                      | 代表                      | 适用                     |
| ------------------ | ------------------------- | ------------------------- | ------------------------ |
| **启动扫描**       | 进程启动时扫描目录        | VSCode, Eclipse, Factorio | 插件目录固定，数量可控   |
| **安装时注册**     | 安装/更新插件时写入注册表 | Chrome, WordPress         | 插件由商店分发           |
| **运行时动态注册** | 运行时通过 API 注册       | JupyterLab, OSGi Services | 需要热插拔               |
| **构建时注入**     | 编译/打包时确定           | 移动端内置模块            | 安全要求高，动态性要求低 |

**推荐**：客户端优先启动扫描（简单可靠），服务端优先配置驱动。

### 6.3 加载策略对比

| 策略             | 时机                  | 优点                       | 缺点               |
| ---------------- | --------------------- | -------------------------- | ------------------ |
| **全量启动**     | 进程启动              | 简单，可用性确定           | 启动慢，内存高     |
| **声明式懒加载** | activationEvents 触发 | 启动快，内存省             | 需要设计激活条件   |
| **事件驱动**     | 事件到来              | 资源利用率高（Chrome MV3） | 需要处理冷启动延迟 |
| **用户触发**     | 用户主动启用          | 最安全（Obsidian）         | 用户感知门槛       |

**推荐**：声明式懒加载是当前最佳平衡。关键原则：**不激活就不加载，不用就不运行**。

### 6.4 通信模式

插件与 Core 之间、插件与插件之间的通信方式：

| 模式                 | 说明                     | 代表                                      |
| -------------------- | ------------------------ | ----------------------------------------- |
| **直接 API 调用**    | 插件调用 Core 暴露的 API | VSCode(`vscode.*`), WordPress 函数        |
| **RPC**              | 进程间远程过程调用       | VSCode Extension Host ↔ 主进程            |
| **消息传递**         | postMessage/sendMessage  | Chrome(content ↔ background ↔ popup)      |
| **事件总线**         | 发布-订阅解耦            | WordPress Hook, Drupal Event              |
| **共享接口**         | 通过 DI 注入服务         | JupyterLab Token, OSGi Services           |
| **remote interface** | 显式声明跨插件 API       | Factorio remote.call/remote.add_interface |

**选型原则**：

- 同进程 → 接口调用或事件总线
- 跨进程 → RPC 或 gRPC
- 跨沙箱 → 消息传递
- 插件间 → 事件总线或显式 remote interface（禁止直接互相 import）

### 6.5 版本兼容

来自 Skyrim 的启发（Form ID 系统 20 年向下兼容）：

- 接口一旦发布，就必须考虑向下兼容
- 新增字段优于修改字段
- 用 manifest 中的 `engines` 字段声明兼容版本范围
- Manifest 版本号可以作为兼容治理手段（类似 Chrome MV2→MV3）

**兼容策略**：

```
插件声明 engines: { host: ">=2.0.0" }
→ 宿主启动时检查 → 不兼容的插件标记为禁用但不删除
→ 提示用户等待插件更新
```

### 6.6 分发机制

| 形态     | 说明                      | 适用         |
| -------- | ------------------------- | ------------ |
| 插件市场 | 中心化分发 + 审核 + 评分  | 开放生态     |
| 本地安装 | .vsix / .zip / 目录复制   | 企业内部分发 |
| 配置驱动 | 通过配置启用/禁用内置模块 | 移动端       |
| Git/URL  | 从 Git 仓库或 URL 安装    | 开发阶段     |
| Workshop | Steam/平台内置分发        | 游戏 Mod     |

## 七、关键教训与收尾

### 7.1 关键教训

| #   | 教训                                                                                      | 来源                     |
| --- | ----------------------------------------------------------------------------------------- | ------------------------ |
| 1   | **不要只用一种扩展机制**——Hook、Strategy、Service、Event 四种模式互补，不同场景用不同机制 | Drupal、Eclipse          |
| 2   | **声明优于编码**——manifest 让框架在加载代码前就知道插件要做什么，不读代码就能做全局决策   | VSCode、Skyrim、Factorio |
| 3   | **懒加载是规模化前提**——20000+ 插件的生态必须靠「不用就不加载」                           | VSCode、Eclipse、Chrome  |
| 4   | **隔离程度决定生态能长多大**——没强隔离 100 个插件互相踩踏，有强隔离 20000 个共存          | Chrome MV3、Eclipse OSGi |
| 5   | **安全模型要默认严格**——默认不安全无人敢用，默认安全用户可自行放宽                        | Obsidian、Chrome MV3     |
| 6   | **版本兼容是生态的生命线**——接口一旦发布就必须考虑向下兼容                                | Skyrim Form ID 20 年兼容 |
| 7   | **移动端 = 受控模块化，不是自由插件化**——包体、启动、审核、权限是硬约束                   | 应用商店审核             |
| 8   | **后端 Mod 化 ≠ 动态加载插件**——先模块化→配置化→服务化→治理化                             | 后端治理实践             |
| 9   | **ToC 产品模块化 ≠ 小程序化**——核心是识别本体 + 抽可变能力，用户侧按场景入口组织          | ToC 产品框架             |
| 10  | **内部办公先做能力插件化，再做 AI Agent 编排**——Agent 需要标准化能力才能稳定工作          | 内部办公阶段实践         |

### 7.2 各端一句话

| 端           | 核心要点                                                              |
| ------------ | --------------------------------------------------------------------- |
| **PC 端**    | Core 稳如磐石，插件按需激活，进程隔离保稳定——参考 VSCode              |
| **Web 端**   | 权限声明即边界，多上下文沙箱即安全——参考 Chrome                       |
| **移动端**   | 受控模块化，配置化运营，复杂能力服务端承载——参考 App Shell + 远程配置 |
| **后端**     | 先模块化→配置化→服务化→治理化，再谈插件化——参考 K8s CRD + SPI         |
| **内部办公** | 统一底座 + 能力插件 + 编排层，系统从入口升级为能力——参考飞书/钉钉     |
| **ToC 产品** | 识别本体不变，能力模块可变，用户无感知，运营可控——参考业务分层        |

### 7.3 一句话总结

> 每个端的插件化都是同一公式的不同实现——先找到本体(Core)，再定义插槽(Slots)，用声明(Manifest)约束能力，用隔离(Host)保护稳定，用 API 解耦交互，用安全机制守住边界。不要从零造轮子，每端都有成熟参照；不要一上来就做大而全，先画本体图→标插槽→选 3 个能力做 PoC 验证闭环。
