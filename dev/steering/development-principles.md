# 开发铁律与基本原则

## 一、开发铁律

### 1.1 代码质量铁律

- **可读性优先**：代码是写给人看的，其次才是给机器执行的
- **简单胜于复杂**：优先选择简单直接的解决方案，避免过度设计
- **DRY 原则**：Don't Repeat Yourself，避免重复代码
- **KISS 原则**：Keep It Simple, Stupid，保持简单
- **YAGNI 原则**：You Aren't Gonna Need It，不要过度设计未来可能用不到的功能
- **SOLID 原则**：面向对象设计的五大原则
  - S: 单一职责原则
  - O: 开闭原则
  - L: 里氏替换原则
  - I: 接口隔离原则
  - D: 依赖倒置原则

### 1.2 安全铁律

- **永远不要信任用户输入**：所有外部输入必须验证和清理
- **敏感信息不入库**：密码必须加密存储，密钥不得硬编码
- **最小权限原则**：只授予完成任务所需的最小权限
- **安全漏洞零容忍**：发现安全问题必须立即修复
- **默认拒绝**：权限控制采用白名单机制
- **深度防御**：多层安全防护，不依赖单一防护措施

### 1.3 协作铁律

- **代码审查必须进行**：所有代码合并前必须经过 Code Review
- **文档与代码同步**：修改代码时必须同步更新相关文档
- **测试覆盖必须达标**：核心业务逻辑测试覆盖率不低于 80%
- **破窗效应零容忍**：发现技术债务及时记录和处理
- **知识共享**：重要决策和技术方案必须文档化
- **及时沟通**：遇到问题及时沟通，不要闷头解决

## 二、通用开发规范

### 2.1 命名规范

#### 基本原则

- **有意义的命名**：避免使用 a、b、temp 等无意义命名
- **避免缩写**：除非是广为人知的缩写（HTTP、URL、ID、DTO）
- **名副其实**：名称应该准确描述其用途
- **可搜索**：使用容易搜索的名称
- **避免误导**：不要使用容易引起误解的名称

#### 语言特定规范

- **Java/C#/Go**：
  - 变量/方法：camelCase
  - 类/接口：PascalCase
  - 常量：UPPER_SNAKE_CASE
  - 包名：lowercase
- **JavaScript/TypeScript**：
  - 变量/函数：camelCase
  - 类/接口/类型：PascalCase
  - 常量：UPPER_SNAKE_CASE
  - 文件名：camelCase 或 kebab-case
- **Python**：
  - 变量/函数：snake_case
  - 类：PascalCase
  - 常量：UPPER_SNAKE_CASE
  - 模块名：snake_case
- **数据库**：
  - 表名：snake_case（小写）
  - 字段名：snake_case（小写）
  - 索引：idx\_字段名
  - 唯一索引：uk\_字段名

### 2.2 注释规范

#### 何时写注释

- **公共 API**：类和公共方法必须有文档注释
- **复杂逻辑**：算法复杂或不直观的代码必须注释
- **业务规则**：特殊的业务规则必须说明
- **临时方案**：TODO、FIXME、HACK 必须标记
- **重要决策**：为什么这样做，而不是那样做

#### 何时不写注释

- **自解释代码**：代码本身已经很清晰
- **废弃代码**：不要注释掉代码，使用版本控制
- **显而易见**：不要写显而易见的注释

```java
// Bad - 显而易见的注释
// 获取用户名
String username = user.getUsername();

// Good - 解释为什么
// 使用缓存避免频繁查询数据库，缓存时间 5 分钟
String username = cache.get(userId, 300);
```

### 2.3 代码组织

#### 单一职责

- **一个类只做一件事**：职责单一，易于理解和维护
- **一个方法只做一件事**：方法名应该准确描述其功能
- **一个模块只负责一个功能**：高内聚低耦合

#### 长度控制

- **方法长度**：不超过 50 行（特殊情况除外）
- **类长度**：不超过 500 行（特殊情况除外）
- **参数个数**：不超过 5 个（考虑使用对象封装）
- **嵌套层级**：不超过 3 层

#### 分层架构

- **严格分层**：禁止跨层调用
- **依赖方向**：上层依赖下层，下层不依赖上层
- **接口隔离**：层与层之间通过接口通信
- **常见分层**：
  - 表现层（Presentation）：UI、API
  - 业务层（Business）：业务逻辑
  - 数据层（Data）：数据访问
  - 基础设施层（Infrastructure）：技术实现

## 三、性能原则

### 3.1 过早优化是万恶之源

- **先保证正确性**：功能正确比性能重要
- **测量后优化**：使用性能分析工具找出瓶颈
- **优化热点代码**：只优化真正影响性能的部分
- **权衡取舍**：性能 vs 可读性 vs 可维护性

### 3.2 常见性能优化

- **数据库**：合理使用索引、避免 N+1 查询、使用批量操作
- **缓存**：缓存热点数据、合理设置过期时间
- **异步**：耗时操作异步处理、使用消息队列
- **资源**：及时释放资源、避免内存泄漏

## 四、错误处理

### 4.1 异常处理原则

- **不要忽略异常**：至少记录日志
- **不要捕获通用异常**：捕获具体的异常类型
- **不要使用异常控制流程**：异常用于异常情况
- **及时释放资源**：使用 try-with-resources 或 finally

### 4.2 错误信息

- **用户友好**：给用户看的错误信息要友好
- **开发详细**：给开发看的错误信息要详细
- **包含上下文**：记录足够的上下文信息
- **可追踪**：使用 TraceId 追踪请求链路

## 五、版本控制

### 5.1 提交原则

- **小步提交**：每次提交只包含一个逻辑变更
- **原子提交**：提交的代码应该是可运行的
- **有意义的提交信息**：清晰描述做了什么改动
- **及时提交**：不要积累太多改动

### 5.2 分支管理

- **主分支保护**：main/master 分支只接受合并
- **功能分支**：每个功能使用独立分支开发
- **及时合并**：功能完成后及时合并，避免长期分支
- **定期同步**：定期从主分支拉取最新代码

## 六、持续改进

### 6.1 代码审查

- **互相审查**：团队成员互相审查代码
- **建设性反馈**：提出问题的同时给出建议
- **学习机会**：把 Code Review 当作学习机会
- **及时响应**：及时响应审查意见

### 6.2 技术债务

- **及时记录**：发现技术债务及时记录
- **定期清理**：定期安排时间清理技术债务
- **权衡取舍**：业务需求 vs 技术债务
- **不要拖延**：小问题及时解决，避免积累

### 6.3 知识分享

- **文档化**：重要决策和技术方案文档化
- **技术分享**：定期组织技术分享会
- **代码注释**：复杂逻辑添加注释
- **团队协作**：鼓励团队成员互相学习
